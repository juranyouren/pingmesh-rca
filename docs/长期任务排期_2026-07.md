# Pingmesh RCA 长期任务排期

更新日期：2026-07-13  
目标：围绕 INFOCOM 论文叙事，依次补齐工程证据、相关工作、ICPTL 方法包装、LLM 幻觉讨论，以及 SECL/SRCL 的有效性实验。

## 1. 总体原则

1. 先补可验证的工程事实，再修改论文主张。
2. 先做失败分桶和离线可复现实验，再进行 prompt 或模型调优。
3. 159 个 case 需要采用分组或时间切分的 train/validation/test；门控阈值不能在全量数据上选择。
4. 内部数据只输出聚合统计，论文和脚本示例不得泄露设备 IP、原始告警描述或 case 标识。
5. SECL 的首要目标是“压缩且不丢关键证据”，SRCL 的首要目标是“降低自动诊断风险”；不能只用 Top-1 单指标评价。

## 3. 工作包拆解

### WP1：工程现象

#### WP1.1 告警时间漂移图示

- 表达目标：同一物理故障沿拓扑传播时，不同设备的告警首次出现时间并不同步；Pingmesh 触发时间是参考点而非所有告警的共同起点。
- 图中必须同时呈现：根因设备、症状设备、拓扑传播方向、真实故障时刻、各设备告警时刻、Pingmesh 触发时刻、时间漂移区间。
- 产出只包含可交给绘图模型/设计师的 prompt，不直接生成图片。

#### WP1.2 聚焦设备数据量统计

- 统计对象：每个 case 经确定性模块选出的 Top-K 聚焦设备。
- 核心指标：每设备 alarm 数、log 数、总事件数、去重告警类型数、描述字符数/估算 token 数；每 case 的总量、均值、中位数、P90/P95/P99、最大值。
- 对照组：全链路全部设备与 Top-K 聚焦设备，量化设备数和证据量压缩比例。
- 分桶：case、设备角色、Top-K 排名、告警量区间；若有安全的标签评估路径，可额外比较根因设备与症状设备，但推理脚本不得读取 `label.json`。
- 输出：机器可读 CSV/JSON 和论文可直接使用的 Markdown 摘要；默认匿名化设备和 case。

#### WP1.3 工程师访谈 TODO

- 需要确认：当前人工定位耗时、可接受目标时间、P50/P90/P95、起止时间定义、复杂 case 的升级条件、误定位与延迟的成本权衡。
- 论文在访谈完成前只写成 TODO，不填入估计数字。

### WP2：FSE 近三年参考文献

- 检索窗口暂定 2023—2026；2026 论文仅纳入已正式公开且元数据可核验的条目。
- 主题优先级：云故障 RCA、LLM/Agent RCA、置信度与拒答、日志/告警语义处理、生产故障诊断。
- 每篇文献标记用途：问题动机、方法对比、LLM 局限、门控理论或实验设计。
- 避免把仅在参考文献中提到 FSE 的论文误判为 FSE 正式论文。

### WP3：设备确定性聚焦模块包装

论文名称采用 **Incident-Conditioned Path-Temporal Localization (ICPTL)**，不再使用含义较泛的 DDFL。

包装主线：

1. Incident conditioning：用 source、sink 和触发时间限定当前故障，而不是对整张图做无条件排序。
2. Two-view evidence：拓扑可行路径支持与体量鲁棒的相对时序支持分别排名。
3. Reliability-aware fusion：根据当前 case 的可观测可靠性融合，而非固定等权。
4. Diagnosability：证据不足或多设备不可区分时显式降低置信度，为 SRCL 提供路由信号。

实现优先级：

| 优先级 | 改进 | 对应失败模式 | 最小实验 |
| --- | --- | --- | --- |
| P0 | 重复告警去重/截尾计数 | 告警量碾压 | raw count vs unique type vs capped count |
| P0 | 相对告警起始时间 | 症状传播造成时间漂移 | absolute onset vs relative onset |
| P1 | source-sink 可行路径走廊 | 全图中心节点偏置 | endpoint bonus vs path support |
| P1 | role/degree 归一化 | Clos 中 Spine/高连接度偏置 | with/without normalization |
| P1 | 可靠性感知融合 | topo/temporal 相互误导 | fixed vs calibrated fusion |
| P2 | 可诊断性评分 | 多候选不可区分 | risk-coverage 曲线 |

### WP4：选择性根因定位中的 LLM 幻觉

需要区分四类可测幻觉：

1. 实体幻觉：输出输入候选集中不存在的设备/IP。
2. 证据幻觉：引用输入中不存在的告警、日志、端口或时间。
3. 因果越界：把相关或下游症状表述为已证实因果。
4. 置信幻觉：弱证据下仍给出高置信、唯一根因。

治理顺序：候选集约束与结构化 schema → 证据 ID 引用 → 决策规则校验 → 弱证据拒答 → 人工复核。评价除 Top-K 外，还需报告 unsupported-claim rate、候选集外输出率、证据引用有效率和 abstention 质量。

### WP5：后两个模块的提升策略

这里将“后两个模块”定义为 SECL 与 SRCL。

#### SECL

- 不再把独立设备摘要器作为准确率提升点；优先实现跨设备可比较的确定性 evidence compiler。
- 利用当前尚未使用的 `alarm_description`，提取 interface、OperStatus、physical link down、reason、时间和严重度。
- 对重复告警做聚类，只保留首发、峰值/最高严重度、持续时间和计数。
- 用 provenance 保证任何摘要项都能回到原始事件。
- 成功标准：显著压缩 token，同时关键证据召回率高且 Top-K 不劣于未压缩输入。

#### SRCL

- 只将“综合分接近、两视图冲突且证据充分”的 case 交给 LLM；双弱 case 直接人工复核。
- 让 LLM 在固定 Top-K 候选内做 pairwise 仲裁，并要求逐项引用证据；不允许自由生成根因设备。
- gate 在 validation 上优化，不以全量 159 case 调阈值。
- 成功标准：在可接受覆盖率下，自动接受 case 的错误率下降；相较 full-LLM 降低调用率、延迟和幻觉率，并在冲突桶内产生净救回。

## 4. 决策门与优先级

### Gate A：W1 结束

如果数据统计不能证明 Top-K 设备仍存在明显的证据膨胀，则将 SECL 从“核心准确率模块”降级为效率模块，论文不再用海量上下文作为强挑战。

### Gate B：W3 结束

只有在 held-out 或至少 validation 消融中，ICPTL 的某项增强稳定改善对应失败桶，才进入主方法；否则保留为分析结果，不包装成贡献。

### Gate C：W5 结束

如果 LLM 在“近分/冲突”桶内的净救回数不为正，则 SRCL 只保留自动接受/人工拒识，不强行保留 LLM 仲裁作为效果贡献。
