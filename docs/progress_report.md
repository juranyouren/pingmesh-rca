## 一、从毕设方案到当前方案：演进路线

### 1.1 毕设方案（v1.0，论文已完成）

```
原始告警 → Modifier 拓扑剪枝(K=10) → LLM 直接推理 → 输出根因 IP
              ↑                              ↑
         Jaccard + PageRank             单轮 CoT prompt
        (无向图 + 静态权重)
```

**核心问题**：
- 数据：104 例，非人工标注（68→146 重新标注后结果大幅变化）
- 剪枝：Modifier 将 376 节点压缩到 10 个，**根因 IP 经常被剪掉**（82/104）
- 推理：单轮 CoT，prompt 中告警/拓扑信息混杂且经常截断
- 评分：只看 `gt_ips[0]`，有多个根因时浪费信息

**实验结果（104 例旧数据，非人工标注）**：

| 方法 | 常规 Top-1 | 常规 Top-3 | 风暴 Top-1 | 风暴 Top-3 |
|------|-----------|-----------|-----------|-----------|
| TraceRCA | 14.44% | 24.44% | 7.14% | 7.14% |
| NetEventCause | 60.00% | 86.67% | 14.29% | 28.57% |
| BiAn | 48.89% | 67.78% | 21.43% | 35.71% |
| 本文方法 | **60.00%** | **92.22%** | **35.71%** | **78.57%** |

### 1.2 当前方案（v2.0，进行中）

```
原始告警 → Skill Pipeline(纯算法) → 证据融合层(压缩) → LLM 重排审核 → 输出
              │                           │                   │
         ┌────┴────────┐         候选设备综合证据表      看告警语义重排
     Skill 1: TopoPR  Skill 3: Temporal    │              "审核而非推理"
     (有向 PageRank)  (Burst 时序)    Top-K 紧凑详情
         │               │
     └────┴── 归一化等权融合 → 综合排名(87%)
```

**核心变化**：

| 维度 | 毕设方案 (v1.0) | 当前方案 (v2.0) |
|------|----------------|-----------------|
| **数据** | 104 例旧标注 | 146 例人工标注 |
| **拓扑** | 无向 PageRank, 剪枝到 K=10 | 有向 PageRank, 全链路(376 节点) |
| **时序信号** | 无 | Skill 3: Burst + Early Bird + Density |
| **算法融合** | 无（依赖 LLM 自行理解） | 归一化等权平均, 综合分 0-100 |
| **LLM 角色** | "推理定位专家"（从零推理） | "重排审核专家"（审核算法排名） |
| **Prompt** | 三 Skill 输出原样拼接, 经常截断 | 证据融合层: 紧凑表 + Top-K 详情, ≤2000 字符 |
| **评分** | 只看 gt_ips[0] | 命中任意 gt_ip 即算命中 |
| **评测** | LLM 结果一个指标 | skill_evaluation / llm_evaluation 分层对比 |

### 1.3 实验数据变化原因

毕设 104 例标注非人工，标注规则可能偏向拓扑结构 → PageRank 消融 38.89% Top-1（虚高）。新数据 146 例为人工运维标注，PageRank 单独降至 14-17%。时序信号才是 DCN 故障中运维人员实际依赖的最强信号（76.22% 单独）。

## 二、当前方案详细介绍

### 2.1 背景与问题

在大型数据中心网络中，Pingmesh 拨测系统会持续监测端到端网络连通性。当出现丢包时，系统会触发告警——但这个告警只告诉我们"哪里丢了包"（sink_ip），不告诉我们"为什么丢包"（根因设备）。运维人员需要从数百台网络设备中找出真正的故障根因。

**三个核心挑战**：
1. **信号异构**：不同设备上有不同种类的告警（BGP中断、端口Down、光模块异常…），没有统一的重要性标尺
2. **拓扑复杂**：CLOS 架构中故障沿上/下游传播，一个根因能触发数十个下游设备同时告警
3. **信息过载**：告警风暴时一台设备可能产生几百条告警，远超 LLM 上下文窗口

### 2.2 整体架构

```
                        ┌─────────────────┐
  146 例人工标注数据 →  │  Skill Pipeline  │
  (Pingmesh 拨测 + 拓扑) │  (纯算法, 无 LLM) │
                        └────────┬────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              ▼                  ▼                  ▼
        Skill 1: TopoPR    Skill 2: Co-Occur   Skill 3: Temporal
        (有向 PageRank)     (告警共现规则)      (时序 Burst 评分)
              │                  │                  │
              └──────────────────┼──────────────────┘
                                 │
                                 ▼
                    ┌─────────────────────┐
                    │   证据融合层         │
                    │   (去重, 归一化,     │
                    │    综合分排序)        │
                    └────────┬────────────┘
                             │
                             ▼
                    ┌─────────────────────┐
                    │   综合证据表          │
                    │   排名|IP|综合分|PR|时序|告警|
                    └────────┬────────────┘
                             │
                             ▼
                    ┌─────────────────────┐
                    │   LLM 重排审核       │
                    │   "告警语义裁决"     │
                    └────────┬────────────┘
                             │
                             ▼
                        最终根因 IP
```

### 2.3 三个 Skill 详解

**Skill 1: 有向 PageRank（topology_pagerank_rank）**

PageRank 在拓扑图上做随机游走，personalization 向量由告警权重和 cross（路径交汇度）初始化。有向图版本利用了故障传播的方向性——故障从上游传到下游，RCA 应该从下游追溯到上游。

$$P(device) = \alpha \cdot \text{personalization}(device) + (1-\alpha) \sum_{v \to device} \frac{P(v)}{out\_degree(v)}$$

**Skill 2: 告警共现规则（co_occurrence_alarm_check）**

从历史错案中挖掘告警组合规则。例如 "BGP邻居中断 + 光模块功率异常" 同时出现时，根因更可能是光模块硬件故障。当前在纯算法消融中贡献微弱（单独 Top-1 ≈ 1-8%），需绑定 topo/temporal 使用。

**Skill 3: 时序嫌疑度（temporal_score_devices）**

三个时序特征，综合为 0-1 的嫌疑度分数：

| 特征 | 公式 | 权重 | 直觉 |
|------|------|------|------|
| Burst Score | `alarms_near_ref_time / total_alarms` | 0.40 | 根因设备告警集中在故障爆发时刻 |
| Early Bird | `1 / rank(first_alarm_time)` | 0.35 | 根因设备告警最先触发 |
| Temporal Density | `alarm_count / active_span` (cap 20/min) | 0.25 | 高密度告警 = burst 行为 |

**为什么时序是最强单信号？** 在 3/4 的 DCN 故障中，根因设备上的告警确实是**最先爆发**且**最集中**的——下游设备收到的影响是延迟且分散的。这不是算法创新，而是数据中反映出来的物理规律。

### 2.4 证据融合层

三个 Skill 的输出如果原样拼接进 prompt，10 节点常规 case 达 6057 字符（是节点数据的 2.4 倍），告警风暴 case 可达 26000+ 字符直接撑爆 LLM 上下文。

**解决方案**：在 Skill 输出与 LLM 输入之间插入融合层：

```
Skill 输出       融合层                           LLM 输入
─────────       ────────                        ────────
topo ranking ─┐
temporal     ─┼─→ 按 IP 合并 ─→ 综合分等权平均 ─→  证据表(一段)
co-occur     ─┘   ↓                              Info 概况(一段)
              去重/归一化                         候选详情(一段)
              ↓
         常规: 6057 → 1526 字符 (-75%)
         风暴: ~26k → 1522 字符 (-93%)
```

证据表格式（LLM 实际看到的）：
```
排名 | IP | 角色 | 综合分 | PR(有向) | 时序(归一化) | Cross | 关键告警
1    | 26.55.35.5 | SPINE | 100.0 | 4.07 | 1.000 | 3 | 端口Down; BGP邻居中断
2    | 26.55.35.4 | SPINE |  88.9 | 4.08 | 0.000 | 1 | —
```

### 2.5 LLM 角色转变

**旧 prompt（毕设）**："你是根因定位专家，请深度推理所有节点数据"

问题：LLM 试图从零理解全网拓扑，忽略告警权重和时序信号，输出大面积空 IP，87% → 33%。

**新 prompt（当前）**："你是排名审核专家，算法已经按综合分排好了。你的任务是在综合分接近时，根据告警名称的语义做出裁决。"

关键约束：
- 不允许引入不在列表里的 IP（防幻觉）
- 综合分差距 > 15 时，默认信任算法
- 综合分接近时，告警语义优先：硬件故障 > 协议中断 > 端口震荡

### 3.1 消融实验：纯算法 Skill 组合 Top-K（146 例人工标注）

**Skill 组合 Top-1 对比**（含 co_occur 的组合全部按其中较优者，权重来源取最优）：

| 等级 | 组合 | Top-1 | Top-3 | Top-5 | 含义 |
|------|------|-------|-------|-------|------|
| **S** | **[1,3] topo+temporal (dir, llm权重)** | **87.41%** | 88.81% | 89.51% | 当前纯算法天花板 |
| S | [1,2,3] all (dir, llm权重) | 86.71% | 88.81% | 88.81% | co_occur 拖后腿 |
| A | [1,3] topo+temporal (undir, llm权重) | 83.22% | 90.21% | 90.91% | 无向稍弱于有向 |
| A | [1,3] topo+temporal (dir, 人工权重) | 81.82% | 88.11% | 88.81% | LLM 权重 > 人工 |
| A | [3] temporal only | 76.22% | 88.81% | 88.81% | 时序是最强单信号 |
| B | [2,3] co+temporal | 75.52% | 87.41% | 87.41% | co_occur 稀释 |
| C | [1] topo only (undir, llm权重) | 17.48% | 23.08% | 32.87% | PageRank 单独很弱 |
| C | [1] topo only (dir, llm权重) | 14.69% | 24.48% | 32.17% | 有向在新数据上更差 |
| D | [2] co_occur only (人工权重) | 7.69% | 16.08% | 19.58% | 告警权重单独无用 |
| D | [2] co_occur only (llm权重) | 1.40% | 3.50% | 3.50% | learn_from_labels 过稀疏 |

**核心发现**：
1. 时序是压倒性最强单信号—3/4 的 DCN 故障中根因设备告警最先爆发
2. 拓扑信号互补：topo 把时序抓不到的 ~13 个 case 补上了（76→87%），+11 pp
3. co_occur(Skill 2) 在等权融合中每加必掉——其噪声稀释了强信号
4. LLM 权重(top1=87.41%) 整体优于人工权重(81.82%)，说明条件概率学习是有效的
5. 有向 PageRank 在组合中生效(dir > undir: 87.41% vs 83.22%)

## 四、未解决问题与风险

1. **LLM 重排是否加分为正？** 当前 87.41% 的算法基线很高，LLM 需要从 89% 的 Top-5 候选中挑对第 1 名。如果"重排"范式不能有效改善，LLM 的角色可能需要降为"仅在综合分接近时裁决"。

2. **告警风暴场景的退化**：旧数据 14 例风暴上 Top-1 从 60% 降到 35.71%（-40%）。新数据 146 例中风暴比例和退化幅度尚无数据。

3. **LLM 前置告警打分的一致性**：同一告警名在不同 case 中出现，LLM 可能给不同分。已通过去重（全局只打一次分）解决，但尚未验证打分的合理性。

4. **co_occur 的价值**：当前在等权融合中拖后腿，但它承载的"告警组合推理"如果能用 LLM 来驱动而非纯规则匹配，可能逆转为加分项。

5. **NIKA 公开数据集**：尚未集成，对跨数据集泛化能力没有数据。
