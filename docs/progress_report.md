## 一、方案演进

### 1.1 毕设方案（v1.0，已弃用）

```
原始告警 → Modifier 拓扑剪枝(K=10) → LLM 直接推理 → 输出根因 IP
```

- 数据：104 例非人工标注
- 核心问题：剪枝经常把根因 IP 踢掉，prompt 超长截断
- 结果不可复现（标注问题）

### 1.2 当前方案（v2.0）

```
Pingmesh 告警 → ┬─ Skill 1: 有向 PageRank ─┐
                └─ Skill 2: 时序嫌疑度    ─┤
                                           ↓
                                    归一化等权融合
                                           ↓
                                    证据融合层（紧凑表）
                                           ↓
                                    LLM 重排审核
                                           ↓
                                      最终根因 IP
```

**核心变化**：

| 维度 | v1.0 | v2.0 |
|------|------|------|
| 数据 | 104 例非人工标注 | 146 例人工标注 |
| 算法 | 无向 PR 单独（38%） | topo+temp 融合（87%） |
| 时序信号 | 无 | Burst + EarlyBird + Density |
| LLM 角色 | 推理定位 | 重排审核 |
| Prompt | 原始拼接，经常截断 | 证据融合层，紧凑表 |
| 评测 | 单一指标 | skill / llm 分层 |

## 二、方案详解

### 2.1 问题

Pingmesh 拨测发现丢包，但不知道是哪个设备坏了。运维人员需要从数百台设备中找出根因。

**两个核心挑战**：
1. **拓扑复杂**：CLOS 架构中一个根因能触发数十个设备同时告警
2. **信号异构**：不同告警种类（BGP中断、端口Down、光模块异常）没有统一的重要性标尺

### 2.2 Skill 1：有向 PageRank

在物理拓扑图上做 personalized PageRank。Personalization 向量由告警权重 + cross 交汇度 + 受影响的端点邻近度初始化。有向图版本反转 linked_from/linked_to 的方向，将随机游走导向拓扑上游。

注意：Spine-Leaf 网络中这些是层次标签而非因果方向，论文应诚实呈现。

### 2.3 Skill 2：时序嫌疑度

三个特征综合评分：

| 特征 | 公式 | 权重 | 含义 |
|------|------|------|------|
| Burst Score | `count(abs(t - ref_time) ≤ 5min) / total` | 0.40 | 告警在故障时刻附近集中度 |
| Early Bird | `1 / rank(first_alarm_time)` | 0.35 | 最早告警的设备排名 |
| Temporal Density | `alarm_count / active_span` (cap 20/min) | 0.25 | 单位时间告警密度 |

参考时间取自 `info["alarm_time"]`（Pingmesh 拨测触发时间）。时间戳 fallback：节点告警/日志 → label.json。

**为什么是最强单信号？** 根因设备的告警最先爆发且最集中。下游设备收到的影响有时间延迟且分散。这是数据反映的物理规律，不是算法创新。

### 2.4 融合与评分

两个 Skill 的得分先各自归一化到 [0,1]，再等权平均得到综合分。与消融实验使用完全相同的评分函数（`skill_pipeline._score_topo` / `_score_temporal`）。

### 2.5 证据融合层

两个 Skill 输出的原始 JSON 如果原样拼接进 prompt，可达 6000+ 字符（常规）到 26000+ 字符（风暴）。融合层压缩为三段紧凑文本：

- **综合证据表**：排名 | IP | 综合分 | PR(有向) | 时序(归一化) | Cross | 关键告警
- **故障概况**：info.json 关键字段
- **候选详情**：Top-K 设备的告警名去重列表 + 拓扑连接

压缩比：常规 −75%，风暴 −93%。大小与告警量解耦。

### 2.6 LLM 角色

旧 prompt 让 LLM "从零推理定位" → 87% 掉到 33%。新 prompt 让 LLM "审核重排算法给出的排名" — 只在综合分接近时用告警语义裁决。不允许编造不在列表里的 IP，不允许因不确定输出空列表。

## 三、实验结果

### 3.1 纯算法消融（146 例人工标注）

| 组合 | Top-1 | Top-3 | Top-5 | 含义 |
|------|-------|-------|-------|------|
| **[1,2] topo+temporal (llm权重)** | **87.41%** | 88.81% | 89.51% | 当前纯算法天花板 |
| [1,2] topo+temporal (人工权重) | 81.82% | 88.11% | 88.81% | LLM 权重 > 人工 |
| [2] temporal only | 76.22% | 88.81% | 88.81% | 时序是最强单信号 |
| [1] topo only | 14-17% | 23-28% | 32% | PageRank 单独很弱 |

**核心发现**：
- 时序 + 拓扑协同：topo 把时序抓不到的 ~13 个 case 补上了（76→87%，+11pp）
