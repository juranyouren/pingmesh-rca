## 0. 现有方案

### 0.1 问题定义

本项目研究数据中心网络（DCN）中由 Pingmesh 丢包/连通性异常触发的根因设备定位问题。Pingmesh 能发现端到端异常，例如 source 到 sink 之间出现丢包，但它天然只给出受影响端点和触发时间，并不直接告诉系统是哪台物理网络设备、端口或链路导致故障。

当前系统的目标是：输入一次 Pingmesh 事件对应的 `info.json`、全链路设备拓扑、设备告警/日志，输出 Top-K 根因设备 IP 列表，并用 Top-1/Top-3/Top-5 命中率评测。

### 0.2 当前主线流程

当前活跃方案可以概括为：

```text
Pingmesh case data
  -> Topology/PageRank ranker
  -> Temporal ranker
  -> Score fusion
  -> Evidence packaging
  -> Trust-tree gate
  -> optional local LLM review
  -> Score_N evaluation
```

核心文件：

| 模块 | 文件 | 作用 |
| --- | --- | --- |
| 拓扑评分 | `Sys/RootCauseAnalyze/skills/topo_ranker.py` | 基于物理拓扑做 Personalized PageRank，结合告警权重、cross、source/sink proximity。 |
| 时序评分 | `Sys/RootCauseAnalyze/skills/temporal_ranker.py` | 计算 Burst、EarlyBird、Density 三类时序特征。 |
| 分数融合 | `Sys/RootCauseAnalyze/skills/fusion.py` | 对 topo/temporal 分数归一化并融合，生成 Top-K 候选。 |
| 证据组织 | `Sys/RootCauseAnalyze/gate/evidence.py` | 将排名、告警、拓扑摘要压缩为 LLM 可读证据。 |
| 信任门控 | `Sys/RootCauseAnalyze/trust_trees/`、`Sys/RootCauseAnalyze/gate/decision.py` | 用可审计规则树决定 bypass LLM、invoke LLM 或 operator review。 |
| 在线推理 | `Sys/RootCauseAnalyze/SkilledAnalyzer.py` | 本地 vLLM 推理、NPU 调度、门控集成、候选摘要。 |
| 评测 | `Sys/Score/Score_N.py`、`Sys/Score/evaluate_trust_gate.py` | 计算 Top-K 指标，输出 gate route 维度统计。 |

### 0.3 Topology/PageRank 评分

拓扑评分把一次故障涉及的网络设备构成图，在图上运行 Personalized PageRank。每个设备的 personalization 主要来自：

- 高危告警权重命中：命中 `all_alarms.json` 中高权重告警时直接加权。
- 告警数量：无高权重命中时，用 `len(alarms) * 2.0` 作为退化信号。
- 日志数量：仅有日志时给弱加分。
- `cross`：设备被多条异常路径交汇时放大其初始分。
- source/sink proximity：设备本身是端点或与端点相邻时给弱加分。

当前实现同时保留 directed/undirected Top-3 诊断信息，用于 topo trust tree 判断该排名是否稳定。

### 0.4 Temporal 时序评分

时序评分不依赖拓扑，只根据设备告警/日志相对 Pingmesh 触发时间的模式打分：

| 特征 | 含义 | 当前权重 |
| --- | --- | ---: |
| Burst | 告警是否集中在故障参考时间 ±5 分钟窗口内 | 0.40 |
| EarlyBird | 设备首条告警在所有设备中的早晚，越早越可疑 | 0.35 |
| Density | 单位时间告警密度，封顶到 20 条/分钟 | 0.25 |

时序得分归一化到 0-1 后参与融合。Temporal trust tree 会检查是否有足够时间戳、Top-1 是否被至少两个子信号支持、排名是否有明显间隔。

### 0.5 融合与证据组织

`fusion.py` 将选中的 skill 分数做统一排序，当前默认 `skills=(1, 2)`，也就是 topo + temporal。融合后的 Top-K 候选进一步被整理成结构化证据：

- `combined_score_rankings`：候选 IP、排名、综合分、设备角色。
- topo/temporal 排名与诊断字段。
- trust tree 状态。
- 候选设备告警名、高危告警、cross、上下游拓扑摘要。
- 故障概况中的 `alarm_time`、`source_ip`、`sink_ip`、`scenario_code`、`alarm_description` 等字段。

这一步的目标是把数百台设备和原始 JSON 压缩成可读、可控、可复核的证据，而不是把全量数据直接塞进 LLM。

### 0.6 Trust-tree Gate

项目已经从旧的连续 confidence score 方案切换为 `trust_tree_v1`。门控输入为 topo trust tree、temporal trust tree，以及两路 Top-K 排名的一致性。

当前路由规则：

| 条件 | 路由 | 含义 |
| --- | --- | --- |
| topo weak 且 temporal weak | operator_review | 两个信号都弱，不自动诊断。 |
| topo 与 temporal Top-1 相同，或 Top-3 overlap >= 2 | accept_combined | 两路信号接近，直接接受融合排名。 |
| topo strong 且 temporal 非 strong | invoke_llm | topo 强但单独不可信，交给 LLM 仲裁。 |
| temporal strong 且 topo 非 strong | accept_temporal | 时序强且 topo 不强，直接接受 temporal。 |
| topo strong 且 temporal strong 但冲突 | invoke_llm | 两路强信号冲突，需要 LLM 仲裁。 |
| 其他不确定情况 | invoke_llm | 交给 LLM 或后续人工分析。 |

这个设计的关键点是：LLM 不再默认参与所有 case，而是只处理冲突、模糊或语义证据可能改变排序的 case；高置信算法 case 可直接绕过 LLM，弱信号 case 进入人工复核。

### 0.7 LLM 角色

项目文档和实验结论都指向同一个判断：LLM 不适合作为无约束的全局重排器。当前 prompt 的设计原则是：

- 默认信任算法综合排名。
- 只在候选设备告警语义提供明确相反证据时调整顺序。
- 只能重排已有候选 IP，不能编造新设备。
- 使用本地 vLLM 和 Ascend NPU，不调用外部 API。

注意：`prompts/skilled.py` 中仍有“146 例 87% Top-1”的旧描述。根据 `docs/数据报告.md` 与 `docs/project_overview.md`，这类高分与历史数据泄漏/数据版本不一致有关，论文正文和后续 prompt 应改用修正后的可信指标。

### 0.8 与已有方法/基线的定位

| 方法类别 | 代表 | 与本项目关系 |
| --- | --- | --- |
| Pingmesh/探测系统 | Pingmesh、R-Pingmesh、RD-Probe、ByteTracker | 提供检测或路径观测能力，本项目接在检测之后做根因设备定位。 |
| 统计/规则 RCA | TraceRCA、FP-Growth、告警关联方法 | 可作为基线；它们偏关联统计，本项目显式引入物理拓扑和时序融合。 |
| 时序点过程 RCA | NetEventCause | 不依赖拓扑，适合拓扑缺失场景；本项目利用 DCN 拓扑结构增强定位。 |
| LLM Agent RCA | BiAn、TAMO、COLA、Confucius | 与本项目同属 LLM-enhanced AIOps，但本项目强调“确定性算法粗筛 + LLM 受控复核 + 门控降成本”。 |
| 多模态/图方法 | ClusterRCA、知识图谱 PageRank、GCN 告警压缩 | 与本项目的图排序思想相近，可作为相关工作支撑。 |

## 2. 挑战、贡献与创新点对应总结

| # | 挑战 | 项目贡献 | 创新点 |
| ---: | --- | --- | --- |
| 1 | Pingmesh 只能检测端到端异常，不能直接定位根因设备；ECMP 和 Spine-Leaf 高扇出会掩盖真实路径。 | 构建面向 Pingmesh 事件的 DCN 根因定位流程，将 source/sink、全链路拓扑、设备告警/日志统一输入到 Top-K 诊断系统。 | 将 Pingmesh 探测事件从“异常检测”扩展为“物理根因设备排序”的完整 RCA 任务建模。 |
| 2 | 仅靠告警数量或简单拓扑邻近容易被告警风暴、下游受害设备和拓扑稀释误导。 | 实现 Personalized PageRank 拓扑 ranker，将告警权重、cross、端点 proximity 与图结构融合。 | 在 DCN 物理拓扑上显式建模故障传播瓶颈，把路径交汇度和告警证据注入 PageRank personalization。 |
| 3 | 单一时序信号不稳定：告警可能缺失、延迟、刷量或由受害设备先触发。 | 实现 Burst、EarlyBird、Density 三特征的 temporal ranker，并与 topo ranker 独立运行。 | 将“早、密、集中”的运维直觉转化为可复现的设备级时序嫌疑分。 |
| 4 | topo 与 temporal 各有盲区，固定依赖任一路都会造成系统性误判。 | 对两路分数归一化融合，输出稳定 Top-K 候选，同时保留每路独立证据供诊断。 | 双路确定性 ranker 互补融合，先把几百台设备缩小到可审查候选集，再进入 LLM/人工层。 |
| 5 | LLM 在 RCA 中容易幻觉、过度重排或破坏已经正确的算法排名；全量调用成本高。 | 将 LLM 限定为候选语义复核器，并用 prompt 约束其默认信任算法、只能重排候选 IP。 | “算法先验 + LLM 受控仲裁”的混合 RCA 范式，避免把 LLM 当无约束黑盒分类器。 |
| 6 | 不是所有 case 都需要 LLM；也不是所有 case 都适合自动诊断。 | 设计 `trust_tree_v1`，根据 topo/temporal 强弱和排名一致性路由到 bypass、LLM 或 operator review。 | 可审计 trust-tree gate：把 LLM 调用从默认路径变为按风险触发的决策路径。 |
| 7 | 原始设备 JSON 很大且噪声多，直接输入 LLM 会造成上下文浪费和不稳定推理。 | 将候选排名、关键告警、设备角色、cross 和上下游摘要打包为紧凑 evidence；支持小模型候选摘要。 | 面向 RCA 的证据压缩层，让 LLM 处理“高密度证据”而非全量原始数据。 |
| 8 | 早期实验出现过 `label.json` 数据泄漏和数据版本不一致，导致虚高结果。 | 明确 label 只在 `Score_N` 评测中读取；推理路径从源码和项目规范上禁止读取 label；提供失败分桶和 gate 评估脚本。 | 将“防泄漏评测”和“失败模式分析”纳入方法论，保证论文实验可解释、可复现。 |
| 9 | 告警权重表覆盖不足，很多告警语义没有进入算法层，PageRank 容易退化为告警数量排序。 | 维护 manual/LLM-learned 权重对比，并在 evidence 中显式暴露 high-severity alarms 给 LLM。 | 把告警语义分成“算法可用权重”和“LLM 可读证据”两层，支持后续逐步增强。 |

## 3. 可能的改进方向

### 3.1 立即建议

1. **修正 prompt 中过期指标**  
   `prompts/skilled.py` 仍写“146 例 87% Top-1”。建议改成不带具体数字的描述，或改为当前可信快照，避免论文实验与线上 prompt 不一致。

2. **补齐 gate 实验主表**  
   当前脚本已经能输出 `gate_effectiveness.json/csv`，建议主实验至少汇报 auto-accept coverage、LLM-call coverage、operator-review coverage、各 route Top-K。

3. **把数据泄漏历史写成威胁与修正**  
   这不是负面信息，反而能强化实验可信度。建议在实验设置中明确：历史高分来自 label 时间戳泄漏；最终结果只使用 label-free inference path。

### 3.2 算法改进

1. **告警描述文本结构化**  
   当前 `alarm_description` 中可能包含端口、Reason、OperStatus、物理链路等信息，但算法层基本没有使用。可抽取端口级、链路级、硬件故障关键词，作为 topo personalization 或 LLM 证据字段。

2. **设备角色感知建模**  
   SPINE/LEAF/CORE 的故障传播模式不同。当前 role 多用于展示，未系统进入评分。可按角色学习不同的传播先验、告警量归一化方式和阈值。

3. **相对时间差建模**  
   现有 temporal 使用绝对时间与全局早晚。可以增加沿拓扑边的 delta time，例如上游首告警到下游首告警的时间差，用于区分根因与受害节点。

4. **source/sink 距离增强**  
   当前 source/sink 只给弱加分。可以从 source/sink 出发做双向 BFS 或路径候选召回，把端到端丢包的拓扑语义转化为距离加权特征。

5. **自适应融合/学习排序**  
   当前融合近似等权。可基于 case 属性动态调节 topo/temporal 权重，例如告警设备比例、时间戳覆盖率、topo/temporal margin、ranker agreement。

6. **告警权重覆盖率分析**  
   对实际出现的 alarm_name 做命中率统计：高频未覆盖告警、低频高危告警、同义告警归一化。先解决覆盖率，再讨论 LLM-learned weights 的收益。

### 3.3 实验改进

1. **超参敏感性分析**  
   PageRank alpha、temporal 时间窗口、Burst/Early/Density 权重、fusion 权重、Top-K 都需要 sweep。若性能对某参数极敏感，论文中应报告稳定区间。

2. **严格 label_v2 评测**  
   `Score_N.py` 已支持更严格的 `label_v2.json` schema。建议主实验尽量使用 primary/secondary root causes，而不是旧 `label.json:top3_ranking`。

3. **跨数据集泛化**  
   当前可信结论主要来自内部单数据集。建议补充 NIKA 或另一个内部 DC 的测试，至少验证趋势：topo/temporal 互补、gate 降成本、LLM 全量重排不稳定。

4. **失败 case 定量归因**  
   继续使用 `analyze_skillpipe_failures.py` 输出 failure buckets，并把“gt 在 Top-3/Top-5/miss Top-5”的分布放入论文，说明哪些错误可被 LLM/调权救回，哪些需要新信号。

### 3.4 系统工程改进

1. **统一文档与脚本命名**  
   `docs/数据报告.md` 中数据路径写法与 `scripts/common.sh` 默认路径存在细节差异，建议统一为 `data/node/nodes_max_labeled`。

2. **清理过时叙事**  
   旧的 SkillBank、continuous confidence、87% 泄漏结果、告警风暴叙事都应标注为 deprecated，避免和当前 trust-tree + built-in skills 主线混杂。

3. **生成论文专用 artifact 清单**  
   建议建立一个 `docs/paper_artifacts.md`，列出每张图、每张表、每个实验命令、每个输出 JSON/CSV 的来源，降低后续写作成本。

## 4. 可直接放入论文的贡献表述草稿

1. 本文提出一种面向 Pingmesh 触发事件的数据中心网络根因定位框架，将端到端丢包告警、物理拓扑、设备告警和日志统一建模为 Top-K 根因设备排序问题。

2. 本文设计了拓扑与时序双路确定性 ranker：拓扑侧通过 Personalized PageRank 融合告警权重、路径交汇和端点邻近信息；时序侧通过 Burst、EarlyBird 和 Density 捕获根因设备的告警触发模式。

3. 本文引入可审计 trust-tree gate，根据 topo/temporal 信号强弱和排名一致性决定自动接受、LLM 仲裁或人工复核，从而降低无效 LLM 调用并避免 LLM 破坏高置信算法结果。

4. 本文构建了 label-free inference 与 label-only evaluation 的实验流程，并通过消融、门控路由统计和失败分桶分析验证各模块作用，避免历史数据泄漏导致的虚高结论。

## 5. 扫描结论

项目当前已经形成比较清晰的论文主线：**Pingmesh 触发的 DCN RCA -> topo/temporal 双路确定性排序 -> evidence packaging -> trust-tree gate -> 受控 LLM 仲裁**。

最适合作为论文核心创新的不是“LLM 直接做 RCA”，而是“确定性网络诊断信号与 LLM 语义复核之间的可审计协同”。实验部分应围绕三个问题展开：

1. topo 与 temporal 是否互补；
2. LLM 是否应该全量参与，还是只在 gate 认为有必要时参与；
3. gate 是否能在少调用 LLM 的同时保持或提升 Top-K 诊断质量。
