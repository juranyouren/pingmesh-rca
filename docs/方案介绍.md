# Pingmesh 网络故障根因定位方案

## 一、背景：我们要解决什么问题

大型数据中心网络中有一个叫 **Pingmesh** 的系统——它持续检测服务器之间的网络连通性。比如服务器 A 到服务器 B 之间的路径上有一个交换机端口坏了，Pingmesh 就能发现"丢包了"，并触发一条告警。

但这条告警只说了一个信息：**"哪里丢包了"**（受影响的端点 IP）。它不告诉你**"是哪个设备坏了"**（根因设备）。

运维人员接到告警后，需要从几百台网络设备中找出是哪一台坏了。这很慢——一个故障可能涉及 300 多台设备同时发出告警，人工逐一排查需要数十分钟。

**我们的目标**：给系统输入 Pingmesh 告警和网络拓扑数据，自动输出最可能是根因的设备 IP 列表。

---

## 二、输入数据是什么

每次 Pingmesh 告警触发后，系统会收集一份数据包，包含：

| 数据 | 说明 | 示例 |
|------|------|------|
| **故障概况** | Pingmesh 检测到的丢包信息 | 源 IP 26.64.72.9 → 目的 IP 29.50.133.135，丢包率 12%，故障时间 2025-12-16 15:46 UTC |
| **设备列表** | 此次故障涉及的所有网络设备 | ~300 台交换机，每台带 mgmt_ip、role（SPINE/LEAF/CORE）、linked_from、linked_to、cross 等 |
| **告警/日志** | 每台设备在此次故障期间发出的告警和日志 | "端口 Down 告警"、"BGP 邻居中断"、"光模块接收功率异常" 等，每条带时间戳 |

这三个数据构成了我们做根因定位的全部输入。

---

## 三、方案思路：两路并行评分 + LLM 复核

**核心想法**：从两个独立的维度给每台设备打分，然后融合、排序，最后由 LLM 做最终确认。

### 3.1 第一路：拓扑评分（看"位置"）

网络设备不是孤立工作的——它们是连在一起的。一个设备的故障会影响它在拓扑上连接的所有下游设备。

我们用 **PageRank** 在物理拓扑上随机游走。简单来说：每台设备有个初始权重（由告警严重程度、路径交汇次数等决定），然后在拓扑上"扩散"——被多条异常路径穿越的设备，分数会越来越高。

**直觉**：交通枢纽位置的设备，如果坏了，影响面最大。PageRank 帮我们找出这种"枢纽"。

### 3.2 第二路：时序评分（看"时间"）

这个思路更简单——不看拓扑，只看告警发生的时间。

根因设备的告警有一个特征：**最早、最集中**。设备坏了 → 立刻发出多条告警 → 然后下游设备受到影响 → 下游设备的告警延迟出现。所以：

- **Burst（集中度）**：告警是否在故障时间点附近密集爆发？权重 0.40
- **Early Bird（最早触发）**：这台设备在这批设备中是第几个发出告警的？权重 0.35  
- **Density（密度）**：单位时间内产生了多少条告警？权重 0.25

三个数加权求和，得到时序嫌疑度。

### 3.3 融合

两路得分各自归一化到 0-1 之间，然后**等权平均**，得到综合分。按综合分排序，取 Top-K 作为候选列表。

---

## 四、LLM 的角色

前两步是纯算法，不依赖大模型，可以独立运行。LLM 的角色是**复核**——检查算法给的排名是否合理。

大模型擅长理解"告警名称的语义"：它知道"光模块硬件故障"比"端口震荡"严重，知道"BGP 邻居中断 + 光模块功率异常"同时出现时，根因大概率是光模块而非 BGP 配置。算法看不懂这些，只会算数。

**LLM 的工作方式**：
- 默认**信任算法的综合排名**
- 只有当候选设备的告警名称提供了明确的相反证据时，才调整顺序
- 不允许编造不在列表里的 IP，不允许输出空列表

---

## 五、为什么这样设计

**1. 两个信号互补**

时序能抓到 ~50% 的根因，拓扑能抓到 ~40%。但它们的覆盖不完全重叠——有时根因设备不是最早告警的（时序错），但它处于拓扑枢纽位置（topo 对）；有时反之。两路融合后到 ~57%，比各自单独用更高。

**2. 算法做粗筛，LLM 做精判**

纯算法把 300 台设备缩到 Top-5 候选，意味着 LLM 只需在 5 个候选中做选择——不是大海捞针，而是 5 选 1。结构化的 JSON prompt 大小 ~1500 字符（远低于 LLM 窗口上限），信息密度高。

**3. 不依赖外部 API**

整个系统在企业内网自托管，使用本地部署的 DeepSeek-32B 模型。脱敏后仍可运行。

## 六、一个具体例子

假设 Pingmesh 检测到 Pod 间丢包 12%，故障概况如下：

- 源 IP: 26.64.72.9
- 目的 IP: 29.50.133.135
- 故障时间: 15:46 UTC

系统拉取了 300+ 台涉及设备后，两个算法分别打分：

- **拓扑评分**：设备 26.55.35.5（SPINE 角色，cross=3）被多条异常路径穿越，PR 得分最高
- **时序评分**：同一台设备 26.55.35.5 的 15 条告警中有 13 条在故障时间 ±5 分钟内，且是所有设备中第一个告警的——时序分也是最高

两路融合后，26.55.35.5 综合分 100，下一名只有 50。差距显著，算法明确指向它。

LLM 复核时看到这个差距，加上设备的告警列表里有"端口 Down"和"链路状态异常"——符合根因特征——确认输出。

最终输出：`["26.55.35.5", "26.55.35.7", "26.55.35.4"]`

## 七、当前效果

在 146 例人工标注的故障案例上评测：

| 阶段 | 说明 | Top-1 准确率 |
|------|------|-------------|
| 纯算法（topo+temporal） | 不依赖 LLM 的自动评分 | ~57% |
| +LLM 复核 | 算法排名 + LLM 语义检查 | ~59%（提升中） |

Top-5 准确率 ~62%（纯算法），意味着在 62% 的故障中，根因设备排在系统输出的前 5 个候选人里——从 300 台设备中缩小到 5 个，大幅降低排查范围。

---

## 八、总结

1. 输入：Pingmesh 告警 + 网络拓扑 + 设备告警/日志
2. 并行计算两个分数：拓扑位置 × 时序模式
3. 融合排序，选出 Top-K 候选
4. 组织为结构化 prompt，交给 LLM 做语义复核
5. 输出最终根因 IP 列表

系统可纯算法独立运行（不依赖大模型），也可启用 LLM 复核获得额外增益。全程在企业内网完成，不依赖外部 API。
