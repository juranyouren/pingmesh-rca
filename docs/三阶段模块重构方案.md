# Pingmesh RCA 三阶段模块重构方案

> 状态：讨论稿，尚未开始代码重构  
> 目的：为后续新窗口继续讨论和实现提供完整上下文

## 1. 背景与重构动机

当前系统的确定性 `pipe` 已经同时包含拓扑排序、时序排序和等权融合，因此论文中的“第一个模块”实际上吸收了大部分有效信号。后续的 Gate 主要负责路由，小模型摘要主要负责压缩输入，二者很难在整体 Top-1 上表现出清晰的独立增益。

当前 159 例实验中的代表性结果如下：

| 方法 | Top-1 | Top-3 | Top-5 |
| --- | ---: | ---: | ---: |
| Topology only | 50.31% | 74.21% | 84.28% |
| Temporal only | 62.89% | 88.05% | 94.34% |
| Topology + Temporal | 76.10%～77.36% | 约 85%～90% | 约 91%～94% |
| Gated cached-summary LLM | 78.62% | 91.19% | 91.19% |

最近一次结果中，确定性 `pipe` 为 77.36%（123/159），加入 Gate、小模型摘要和 LLM 后为 78.62%（125/159），整体只净增加 2 个正确案例。这并不表示后续机制没有价值，而是当前模块边界使大量准确率贡献都被计入了第一个模块。

因此，本方案不是通过削弱现有方法人为制造消融增益，而是重新划分职责，让每个阶段回答不同的问题：

1. 根因可能位于哪里？
2. 哪个候选设备具有更充分的故障证据？
3. 当前结论是否足够可信，是否需要进一步推理？

## 2. 总体方案

新的系统分为三个阶段：

```text
Pingmesh Case
    ↓
第一模块：故障感知的拓扑图聚焦
    ↓
第二模块：时序—语义证据收集与初步排序
    ↓
第三模块：决策可信度分类与选择性语义推理
    ├─ 可信：直接输出初步排名
    ├─ 冲突：主 LLM 语义仲裁
    └─ 证据不足：扩大候选或人工复核
```

推荐的论文叙事为：

> 首先利用受故障端点约束的物理拓扑聚焦可能的故障传播区域；随后收集区域内候选设备的时序和告警语义证据，生成初步根因排名；最后根据证据完整性与一致性判断结果能否直接采用，仅对冲突案例执行受约束的语义推理。

该叙事不再把 Topology 和 Temporal 描述成两个并列 Ranker。Temporal 被封装在第二模块的证据收集与初排过程中，并与小模型语义提取共同构成“时序—语义证据增强”。

需要强调：这里是对时序能力进行模块内聚，而不是在方法和实验中隐藏时序。论文仍需公开时序特征定义，并通过内部消融区分时序与小模型各自的贡献。

## 3. 第一模块：故障感知的拓扑图聚焦

### 3.1 研究问题

> 给定 Pingmesh 异常端点，根因设备可能位于哪些拓扑区域？

### 3.2 职责

第一模块只负责构造高召回的拓扑候选空间，不负责最终根因排序。建议使用：

- source/sink 条件化；
- 可行路径或路径走廊；
- cross-path 覆盖；
- 与异常端点的距离；
- 图中心性；
- 设备角色和上下游关系。

第一模块原则上不应使用：

- 告警数量；
- 告警语义权重；
- 日志数量；
- 告警时间特征。

当前 `score_topo()` 的 PageRank personalization 同时使用了告警权重、告警数量和日志，因此后续实现时应将这些非拓扑信息移入第二模块，形成更清晰的职责边界。

### 3.3 输出

建议输出稳定的结构化结果：

```json
{
  "candidate_ips": [],
  "topology_features": {},
  "feasible_path_evidence": {},
  "candidate_recall_scope": 10
}
```

### 3.4 关键约束

不能直接把当前 Topology Top-5 当作后续唯一候选集合，因为当前 Topology Top-5 召回率只有 84.28%。一旦根因被第一模块过滤，后续小模型和主 LLM 都无法救回。

建议使用以下方案之一：

- Topology Top-10 或 Top-20；
- 所有位于可行路径走廊内的设备；
- Topology Top-K 加端点邻域扩展。

第一模块的首要指标应是 Candidate Recall@K，目标建议达到 95% 以上；Top-1 仅作为辅助指标。

## 4. 第二模块：时序—语义证据收集与初步排序

### 4.1 研究问题

> 在拓扑候选设备中，哪些设备具有更直接、更早且更具体的故障证据？

### 4.2 内部组成

第二模块由三个内部步骤组成：

1. 确定性证据收集；
2. 小模型结构化语义提取；
3. 时序—语义联合初排。

#### A. 确定性时序统计

精确数值由程序计算，不交给小模型计算：

- first event offset；
- relative onset rank；
- burst；
- 去重后的 temporal density；
- duplicate ratio；
- 相邻设备之间的 relative lag；
- 故障窗口内和窗口外的事件比例。

示例：

```json
{
  "first_event_offset_ms": -12400,
  "relative_onset_rank": 1,
  "burst_score": 0.82,
  "density_score": 0.36,
  "duplicate_ratio": 0.71
}
```

#### B. 小模型结构化语义提取

小模型不再定位为“一句话摘要器”，而是“结构化语义证据提取器”。

建议输入：

- `alarm_name`；
- `alarm_description`；
- 接口名称；
- AdminStatus/OperStatus；
- 精确事件时间戳；
- 设备角色；
- 原始证据编号；
- 程序计算的时序统计。

建议输出严格 JSON：

```json
{
  "event_type": "physical_link_down",
  "object": "100GE1/1/25",
  "state_change": "up_to_down",
  "locality": "local",
  "specificity": "direct",
  "evidence_ids": [0, 1]
}
```

小模型必须禁止：

- 直接判断某设备是否为根因；
- 输出候选排名；
- 编造输入中不存在的接口、设备或事件；
- 自行计算精确时间差和告警数量；
- 输出隐藏思维过程。

所有小模型字段必须能回指原始证据，未成功解析时使用 `unknown`，不能自动补全。

#### C. 初步排序

程序将以下信号组合成初步排序：

```text
preliminary_score =
    topology_prior
  + temporal_evidence
  + semantic_specificity
  + temporal_semantic_consistency
```

不建议继续对两个最大值归一化分数简单求平均。可以考虑：

- rank percentile；
- weighted reciprocal rank fusion；
- 根据数据完整度和证据可靠性动态分配权重；
- 保留拓扑候选并集，避免高分候选挤出正确设备。

第二模块建议输出：

```json
{
  "evidence_table": [],
  "preliminary_scores": {},
  "preliminary_ranking": [],
  "evidence_quality": {}
}
```

## 5. 第三模块：决策可信度分类与选择性语义推理

### 5.1 命名说明

此前讨论中的“执行度分类”如果表示判断结果能否直接采用，建议在论文中使用更明确的名称：

> 决策可信度分类与选择性语义推理

也可以称为“结果可执行性分类”，但需要明确其本质是对自动决策风险进行分类，而不是预测一个未经校准的连续置信分数。

### 5.2 决策类别

Gate 将案例分成三类：

```text
reliable      → 直接采用初步排名
conflicting   → 调用主 LLM 进行语义仲裁
insufficient  → 扩大候选或转人工复核
```

分类依据可包括：

- Top-1/Top-2 排名间隔；
- 拓扑先验、时序证据和语义证据的一致性；
- 时间戳覆盖率；
- 告警描述覆盖率；
- 是否存在直接物理状态变化；
- 候选之间是否存在强证据冲突；
- 候选空间是否充分。

### 5.3 主 LLM 的职责

主 LLM 不执行全量自由排序，只处理 `conflicting` 案例，并且：

- 只能在合法候选集合中输出；
- 默认保持第二模块的初步排名；
- 只有明确、可核验的相反证据才能调整顺序；
- 输出 supporting evidence 和 counter evidence；
- 证据不足时显式返回 insufficient evidence。

已有 38 个 LLM 路由案例中，`strong_ranker_conflict` 是最适合保留的 LLM 路由：

| 方法 | 该子集 Top-1 |
| --- | ---: |
| LLM | 68.75% |
| Topology | 50.00% |
| Temporal | 37.50% |

`topo_strong_defer_to_llm` 不应继续无条件调用 LLM，应先使用第二模块的语义证据判断是否存在反证。双弱或候选召回不足的案例也不应直接依赖 LLM，而应扩大候选或人工复核。

## 6. 模块边界与数据流

三个模块的输入输出应在实现中明确记录：

| 模块 | 输入 | 输出 |
| --- | --- | --- |
| 拓扑图聚焦 | endpoints、物理拓扑 | `candidate_ips`、拓扑特征 |
| 证据收集与初排 | 候选设备、告警、日志、时间戳 | evidence table、初步排名 |
| 可信度分类与推理 | 初步排名、证据质量、候选证据 | route、最终排名、可核验解释 |

最终 `res.json` 建议保存每个阶段的中间结果：

```text
topology_candidates
topology_features
temporal_features
semantic_features
preliminary_ranking
gate_route
final_ranking
```

这样可以逐案例判断每个模块修复了什么、破坏了什么，而不是只比较最终百分比。

## 7. 推荐消融设计

### 7.1 主模块累积消融

| 实验 | 内容 | 回答的问题 |
| --- | --- | --- |
| M1 | 拓扑图聚焦 | 拓扑候选能力 |
| M1+M2 | 增加证据收集与初步排序 | 时序—语义精排贡献 |
| Full | 增加可信度分类与选择性语义推理 | 最终仲裁贡献 |

预期使用现有结果形成如下叙事，但最终数字必须重新实验确认：

```text
M1                          约 50%
M1 + M2                     约 76%～77%
M1 + M2 + M3                当前观察到 78.62%
```

### 7.2 第二模块内部消融

| 实验 | 配置 |
| --- | --- |
| M2-full | Temporal + Small Model + 联合初排 |
| M2 w/o temporal | 移除确定性时序特征 |
| M2 w/o small model | 只保留时序和确定性事实 |
| M2 w/o semantic consistency | 保留语义，但不建模时序—语义一致性 |

这组实验用于证明第二模块的增益不是完全来自 Temporal，小模型确实提供了独立语义信息。

### 7.3 第三模块内部消融

| 实验 | 配置 |
| --- | --- |
| Full | Gate + selective LLM |
| w/o Gate | 所有案例调用 LLM |
| w/o LLM | Gate 只能选择初步排名或人工复核 |
| Oracle Gate | 离线分析理论路由上限，不进入推理路径 |

### 7.4 小模型固定集合消融

所有小模型对比必须固定相同候选、相同 Gate 路由案例、相同主模型参数和随机种子：

| 实验 | 主 LLM 输入 |
| --- | --- |
| A0 | 原始设备事实 |
| A1 | 程序确定性无损压缩事实 |
| A2 | 无损事实 + 小模型结构化语义 |

核心比较是 `A1 → A2`，因为它隔离了小模型新增语义的贡献。`A0 → A1` 主要衡量证据组织和压缩效果。

## 8. 评测指标

不能只报告 Top-1/Top-3/Top-5，还应报告：

- Candidate Recall@K；
- MRR；
- `fix`：加入模块后由错变对的案例数；
- `harm`：由对变错的案例数；
- `net_gain = fix - harm`；
- `unique_win`：只有某模块或专家命中的案例数；
- Gate coverage 和 selective accuracy；
- LLM corrected/corrupted/unchanged/all-miss；
- prompt token 数；
- 压缩率；
- 小模型事实遗漏率和幻觉率；
- 摘要/提取缓存生成时延；
- 主 LLM 调用比例和每修复一个案例的调用成本。

159 个案例中，一个案例约等于 0.63 个百分点。77.36% 到 78.62% 实际只是增加 2 个正确案例，因此结果应同时报告案例数，并使用 McNemar 检验或 paired bootstrap 置信区间，避免把 1～2 个案例波动描述成稳定提升。

## 9. 实验公平性要求

- 推理路径不得读取 `label.json`；
- 所有阈值和融合权重只能在训练/开发划分上确定；
- 最终测试集冻结模块、候选范围、Gate 规则和 prompt；
- 相同消融必须使用相同案例和分母；
- LLM 对比必须使用相同主模型参数、温度和随机种子；
- 小模型实验必须使用相同缓存版本和证据组织版本；
- 不得使用旧 143 例、旧 Skill ID 的 `full_ablation/summary.json` 与当前 159 例结果混合比较。

## 10. 当前实现中必须先处理的问题

### 10.1 Skill 消融没有真正隔离输入

`SkilledAnalyzer._build_final_prompt()` 接收 `selected_skill_ids`，但当前 `build_fused_evidence()` 内部硬编码：

```python
skill_ids=(1, 2)
```

因此使用 `SkilledAnalyzer.py -s 1` 和 `-s 2` 进行 LLM 路径消融时，两组仍可能获得相同的 Topology + Temporal evidence。这类结果不能用于论文结论。

重构时必须将启用的组件显式传递给证据构建器，并在结果中记录实际启用的 evidence sources。

### 10.2 Gate 评测缺少实际融合基线

当前 `evaluate_gate_selection.py` 主要比较 Topology、Temporal 和 LLM，没有把 `combined` 纳入逐案例对照；同时 `llm_win_rate` 把 tied-for-best 也计入 win。

后续评测必须针对同一案例统计：

```text
LLM corrected combined baseline
LLM corrupted combined baseline
LLM unchanged
all methods miss
```

### 10.3 不同策略不能使用不同分母

Gate policy 消融必须保证所有策略在同一组案例上得到完整输出。`llm_unavailable`、operator review 或空输出不能通过缩小有效样本分母来使某个策略看起来更准确。

## 11. 建议实施顺序

### 阶段 0：逐案例贡献与理论上限分析

在修改算法前先生成统一的案例归因表，记录：

- Topology 是否命中；
- Temporal 是否命中；
- 当前融合是否命中；
- LLM 是否命中；
- GT 是否在各候选集合中；
- 每个模块的 unique win；
- Oracle expert selection 上限。

如果某个模块没有 unique win，说明它当前没有提供互补信息，应优先修改特征，而不是调整消融顺序。

### 阶段 1：拆分拓扑聚焦与证据初排

- 将当前 Topology scorer 中的告警和日志信号移入第二模块；
- 构造高召回的拓扑候选空间；
- 保留当前 Legacy pipe 作为性能对照；
- 验证 M1 与 M1+M2 的真实累计增益。

### 阶段 2：将摘要器改为结构化语义提取器

- 增加 `alarm_description`、接口状态和证据编号；
- 使用严格 JSON schema；
- 缓存只保存最终结构化输出，不保存 `<think>`；
- 评估事实保真率、解析成功率和小模型独立净增益。

### 阶段 3：实现可信度分类和选择性推理

- 根据第二模块的 evidence quality 设计 Gate；
- 重点保留 strong conflict LLM 路由；
- 拆分 topo-strong、unresolved 和候选不足案例；
- 只在开发集调整规则，测试集冻结。

### 阶段 4：完整消融和统计检验

- 运行主模块和内部组件消融；
- 输出逐案例 fix/harm/net gain；
- 报告准确率、召回、覆盖率、成本和置信区间；
- 根据结果决定小模型是准确率模块还是效率模块。

## 12. 下一窗口需要继续讨论的问题

1. 第一模块的候选空间具体采用 Top-K、路径走廊还是二者结合？
2. 第一模块是否完全移除告警权重，还是保留最小的故障端点种子？
3. 第二模块的小模型 JSON schema 最终包含哪些字段？
4. `alarm_description` 在当前数据中的实际覆盖率和质量如何？
5. 时序—语义联合初排使用规则分数、RRF 还是可校准模型？
6. Gate 的三个类别是否命名为 reliable/conflicting/insufficient？
7. 人工复核案例在 Top-K 准确率统计中如何计入，避免不同分母？
8. 数据集采用固定 train/dev/test 还是 grouped cross-validation？
9. 如何定义小模型模块的成功标准：准确率提升、非劣效还是效率收益？
10. 是否先实现阶段 0 的统一归因脚本，再开始实际算法重构？

## 13. 当前推荐结论

推荐将论文与系统统一为以下三模块：

1. **故障感知的拓扑图聚焦**：生成高召回拓扑候选空间；
2. **时序—语义证据收集与初步排序**：确定性时序统计与小模型语义提取共同完成候选精排；
3. **决策可信度分类与选择性语义推理**：Gate 判断是否可直接采用，主 LLM 只仲裁冲突案例。

这一结构能够形成清晰的“候选空间 → 证据精排 → 风险控制与语义仲裁”递进关系，同时避免把全部准确率贡献塞入第一个模块。最终是否能够形成稳定的 `约 50% → 约 76%～77% → 约 79%` 累积结果，需要在修复消融隔离问题并完成逐案例归因后重新验证。
