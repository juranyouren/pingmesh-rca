一、Pingmesh 体系与网络探测层 (6篇) — 论文方法的技术基础

1. Pingmesh (Guo et al., SIGCOMM 2015)
   方法: 数据中心全网状 TCP/HTTP 端到端延迟测量系统, 通过 Pingmesh Agent
         在每台服务器上定期探测所有其他服务器, 生成全网延迟矩阵
   贡献: 首次实现 DCN 任意两服务器间任意时刻延迟测量
   与本文关系: ★★★★★ 本文方案的触发源, 直接基于 Pingmesh 丢包告警

2. R-Pingmesh (Liu et al., SIGCOMM 2024)
   方法: 面向 RoCE 网络的服务感知监控, 基于商用 RNIC 精确测量 RTT
         和端主机处理延迟, 区分 RNIC 丢包与网络丢包
   贡献: 85% 的问题定位准确, 157个交换机网络问题全部准确
   与本文关系: ★★★★ RoCE 场景的 Pingmesh 演进, 本文方法可直接适配

3. RD-Probe (Ding et al., SIGCOMM 2024)
   方法: 大规模虚拟化 DCN 底层网络黑盒探测, 结合随机+确定性两种策略
   贡献: 二层网络覆盖率 80.9%→99.5%, 华为云3区域部署
   与本文关系: ★★★ 探测层互补, 与本文关注的故障定位形成上下游

4. ByteTracker (Guo et al., SIGCOMM 2025)
   方法: 无需终端 Agent 的实时路径感知探测, 集中式探测器+交换机报文镜像
   贡献: 5秒内故障检测, 定位精度接近 100%
   与本文关系: ★★★ 更轻量的探测方案, 但覆盖完整性不如 Pingmesh 全网状

5. deTector (Peng et al., USENIX ATC 2017)
   方法: 拓扑感知的主动探测, 通过贪心算法选择探测路径实现丢包定位
   贡献: 仅基于端到端观测即可定位, 无需 traceroute 等辅助工具
   与本文关系: ★★★ 探测路径选择与本文拓扑剪枝有相似思想

6. Classifying Host-Side RDMA Pingmesh (Ding et al., ICPADS 2025)
   方法: LSTM 预训练+微调, 对 RDMA Pingmesh 结果进行细粒度故障分类
   贡献: 跨拓扑的小样本传输层故障分类
   与本文关系: ★★ 与本文互补——本文做定位, 该文做分类

二、网络 RCA 核心方法 (11篇) — 本文的核心比较对象

7. BiAn (Wang et al., SIGCOMM 2025)
   方法: 三阶段 LLM 多智能体框架——①信息抽取 Agent 结构化监控数据,
         ②单设备分析 Agent 按 SOP 评估异常度, ③全局验证 Agent 交叉评分
   贡献: 生产部署10个月, 根因定位时间缩短 20.5%, 高风险事件缩短 55.2%
         准确率较基线提升 9.2%
   与本文关系: ★★★★★ 直接基线。本文三阶段(拓扑剪枝→LLM推理→Skill闭环)
         与 BiAn 三阶段(抽取→单机→全局)形成对比。BiAn 依赖 LLM 全程,
         本文在 LLM 前用拓扑统计做预筛选, 告警风暴场景有优势

8. NetEventCause (Yuan et al., TNNLS 2025)
   方法: 多变量神经时序点过程 (ERPP: GRU + Attention + 指数衰减核),
         通过积分梯度归因计算 root_alarm_prob = prior / conditional intensity
   贡献: 无需拓扑, 华为神农 IMOC 平台部署 (20万+实体), 超越 SOTA TPP 模型
   与本文关系: ★★★★★ 直接基线。NEC 完全不需要拓扑, 本文利用拓扑信息
         互补——NEC 在拓扑缺失时有用, 本文在拓扑可用时更精确

9. TraceRCA (Li et al., IWQoS 2021 / TSC 2021)
   方法: ①异常 Trace 检测 → ②FP-Growth 频繁项集挖掘可疑服务集 →
         ③Jaccard 相似度 + 传播方向推断排序
   贡献: Top-1 较无监督 SOTA 提升 44.8%, 某大型商业银行生产部署
   与本文关系: ★★★★★ 直接基线。映射: trace→Pingmesh路径, service→网络设备。
         本文在此基础上引入 LLM 语义推理

10. SkyNet (Yang et al., SIGCOMM 2025)
    方法: 多源监控数据统一输入格式 → 告警分组 → 严重度评估 → 过滤非关键告警
    贡献: 阿里云生产部署1.5年零漏报, 80%+ 故障的止损时间缩短
    与本文关系: ★★★★ 告警风暴场景直接相关。SkyNet 做告警聚合降噪,
          本文做根因排序, 两者可组合

11. SkeletonHunter (Liu et al., SIGCOMM 2025)
    方法: 针对容器化大模型训练网络的诊断, 利用 overlay/underlay 关联分析
    贡献: 高动态容器环境下的快速故障定位
    与本文关系: ★★★ 场景不同(容器训练 vs 传统 DCN), 但多层关联思想可借鉴

12. Hawkeye (Wang et al., SIGCOMM 2025)
    方法: PFC 溯源——细粒度 PFC 感知遥测 + 网内因果追踪 + 溯源诊断算法
    贡献: 90%+ 精确率, 开销较基线低 1-4 个数量级
    与本文关系: ★★ RDMA 特定场景, 但因果追踪思想有参考价值

13. Canary (Chen et al., TON 2026)
    方法: 部分流量监控 + 数据包路径信息编码, 检测并定位静默丢包/损坏
    贡献: 内存开销降低 97%, 故障链路 F1 提升 20%+
    与本文关系: ★★★ 同样是丢包定位但技术路线不同(数据面 vs 控制面)

14. NetDx / Model-Based Diagnosis (Wu et al., 2025)
    方法: 基于数据包转发和路由模型的系统化诊断推导, 覆盖硬件/固件/软件故障
    贡献: 故障注入 100% 诊断正确, 33例云厂商故障中30例秒级诊断
    与本文关系: ★★★★ 白盒模型驱动的方法, 与本文 LLM 黑盒方法形成对比

15. FaultInsight (Bi et al., KDD 2024)
    方法: 深度因果发现——异构主机级指标的因果图构建, 多视角故障诊断
    贡献: 字节跳动生产环境部署, 较 SOTA 因果发现方法显著提升
    与本文关系: ★★★ 因果发现方法, 本文用 PageRank 替代了因果推断步骤

16. AmocRCA (Altenbernd et al., FSE 2025)
    方法: AMOC(At Most One Change)分割 + 相对相关性排序, 仅用指标数据
    贡献: 无需拓扑/图谱/日志/追踪, 适合大规模快速定位
    与本文关系: ★★★ 极简路线——只用指标数据。本文用更丰富的拓扑+告警信息

17. ClusterRCA (Gu et al., 2025)
    方法: 融合拓扑+性能指标+系统日志三种模态, 故障传播图+定制随机游走
    贡献: HPC 场景 Avg@5=0.9962, 较单一模态基线提升约 0.378
    与本文关系: ★★★★ 多模态融合+随机游走定位, 与本文 PageRank 思路接近

## 本文方法定位

现有方法在以下维度存在不足 (本文的差异化贡献):

1. **拓扑利用不充分**: TraceRCA 只用共现统计, NEC 完全忽略拓扑, BiAn 的
   拓扑信息仅作为 LLM 输入文本——本文用 PageRank 在拓扑图上显式建模
   故障传播, 是拓扑利用最充分的方案

2. **LLM 与统计方法割裂**: COLA/BiAn 将 LLM 作为独立模块, 不反馈统计层——
   本文的 Skill 闭环 (LLM反思→生成Python插件→热加载执行) 实现了
   LLM 知识向统计层的迁移

3. **告警风暴场景被忽视**: 大多数方法在 ≤600 条告警的常规场景评测,
   本文显式区分常规/风暴场景并针对性优化

4. **缺乏可执行知识沉淀**: 本文 SkillBank 将 LLM 推理结果编译为可复用
   的 Python 诊断插件, 实现了从"推理"到"自动化"的闭环

## 2026-07-05 补充：信任门控与 LLM 角色

新增独立调研文档：`docs/文献调研补充_信任门与LLM角色.md`。

补充结论：

1. 本文 Trust-Tree Gate 可放入 selective prediction / reject option / abstention 的研究脉络中，核心评价应同时看自动覆盖率和自动输出错误率。
2. COLA 是最接近的直接类比：先用统计相关挖掘处理高置信样本，只把低置信或不确定样本交给 LLM reasoning。本文将该范式从告警聚合扩展到设备级 Top-K RCA。
3. LM-PACE 说明 RCA 推荐本身需要置信估计，本文则用 topo/temporal trust tree 替代 LLM 自评置信度。
4. RCACopilot、RCAgent、TAMO、BiAn 证明 LLM 适合做运维语义理解、工具调用、证据组织和 operator assistance。
5. OpenRCA 与 LLM reasoning failure studies 说明 LLM 在复杂 RCA 中仍存在长上下文、多跳因果和证据选择不稳定，因此本文不应把 LLM 写成主排序器，而应写成受门控约束的仲裁器。
