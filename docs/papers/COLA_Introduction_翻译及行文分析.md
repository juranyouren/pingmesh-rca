> 论文: Knowledge-aware Alert Aggregation in Large-scale Cloud Systems: a Hybrid Approach
> 发表于 IEEE/ACM ICSE-SEIP 2024, 港中文 + 华为云

## 一、ABSTRACT 翻译

在大规模云系统中，一次系统故障会触发"告警风暴"（alert storm），即大量相互关联的告警同时爆发。尽管这些告警可追溯至少数几个根因，但其数量之庞大使得人工逐一处理不可行。因此，告警聚合对于帮助工程师聚焦根因、加速故障解决至关重要。现有方法通常采用基于语义相似度或基于统计的方法来聚合告警。然而，基于语义相似度的方法忽略了告警之间的因果逻辑，而基于统计的方法难以处理低频告警。

针对这些局限，我们提出引入外部知识——即告警的标准操作程序文档（Standard Operation Procedure, SOP）作为补充。我们提出 COLA，一种融合关联规则挖掘与大语言模型推理的混合告警聚合方法。关联挖掘模块有效捕捉告警之间的时间和空间关系，以高效的方式度量其关联程度；随后，仅将置信度较低的不确定告警对转发至 LLM 推理模块进行深度分析。该混合设计兼具统计方法对高频告警的证据积累优势和 LLM 对复杂案例的推理能力，同时保证了 COLA 在实际场景中处理大量告警的整体效率。

我们在某大规模云平台的生产环境中收集了三个数据集，对 COLA 进行评估。实验结果表明，COLA 的 F1 分数达到 0.901 至 0.930，显著优于现有最先进方法，且保持了可比的效率。我们还分享了在 Cloud X 生产环境中部署 COLA 的实践经验。

## 二、Section 1: INTRODUCTION 翻译

### 2.1 背景与重要性

云系统（如 Microsoft Azure、Amazon AWS、Google Cloud Platform）为全球客户提供关键服务。确保这些系统的可靠性对于避免用户不满和经济损失至关重要。例如，Amazon 在 Prime Day 期间每小时的宕机成本高达 1 亿美元 [2]。

在实践中，云系统中的故障（如意外中断或服务降级）是不可避免的 [2, 4, 18]。为了及时发现和处理故障，现代云系统配置了全面的监控机制，持续监测云服务的健康状态，产生包括 KPI（关键性能指标）、日志和调用链在内的监控数据 [13, 22, 23, 26, 27, 33]。当在这些数据上检测到异常模式时，系统会触发告警，通知值班工程师（On-Call Engineer, OCE）进行检查和处置。

### 2.2 告警风暴问题

当故障发生时，大量相互关联的告警会在短时间内集中爆发——这一现象通常被称为臭名昭著的"告警风暴"问题 [39]。该问题主要源于现代云系统的大规模和复杂依赖关系。具体而言，一个云系统可能拥有并监控数十至数百个服务 [7, 8]。此外，这些服务之间相互依赖，意味着一个服务的故障有可能传播至其他服务，这一过程被称为级联故障（cascading failures）[15]。例如，数据库服务中发生的故障可能影响所有依赖它的服务，如用户认证、交易处理或数据分析。在此过程中，告警通常以无协调的方式被触发，令值班工程师难以手动应对。因此，自动地将由同一故障（即具有相同根因）触发的告警进行聚合，对于高效处置故障至关重要。

### 2.3 现有方法的局限性

已有大量工作致力于告警聚合，可分为基于语义相似度的方法和基于统计的方法两类。**基于语义相似度的方法**，如 AlertStorm [39]、LiDAR [6] 和 OAS [3]，利用自然语言处理技术（如 word2vec 和 BERT）比较两条告警的语义相似度来判定关联。然而，具有因果关联的告警在语义上可能截然不同。例如，一条"服务器过载"告警可能与一条"数据库响应变慢"告警存在因果关系，尽管两条告警的文本语义差异悬殊。另一方面，**基于统计的方法**，如 Warden [24]、LiDAR [6] 和 iPACK [25]，从告警的共现模式中学习其关联关系。然而，此类方法依赖大量历史数据进行学习，对于从未见过或出现频率极低的告警难以给出准确判断。

### 2.4 本文思路：引入 SOP 知识 + LLM 推理

针对现有研究的局限，受大语言模型在复杂推理任务上取得的最新进展 [1, 5, 19] 的启发，我们提出利用 LLM 进行大规模云系统中的告警聚合。其核心直觉是：通过从已有告警知识中学习，让 LLM 对两条告警之间的关联关系进行推理。

具体来说，在 Cloud X 中，每条告警都关联一份全面的文档，称为标准操作程序（SOP）。这些 SOP 包含有关告警的详细信息，如触发条件、严重级别、潜在影响、可能的根因和推荐的处置步骤。运维工程师在维护活动中持续更新 SOP（例如当他们观察到新的潜在影响时）。这些 SOP 揭示了告警背后的因果逻辑，使我们能够进行基于知识的推理，而非仅仅依赖语义相似度比较。此外，LLM 能够为现场工程师生成可解释的结果，帮助他们理解聚合结论。

### 2.5 面临的三个挑战

然而，设计这样的方案面临以下三个挑战：

**挑战一：缺乏领域知识。** 现有 LLM 并非为告警聚合任务专门设计。这些模型通常使用公开的通用知识源（如 Wikipedia 和 Github）进行训练，因此缺乏关于云系统中告警的特定知识，而这些知识对于准确推理告警间的关系至关重要。

**挑战二：长文本理解困难。** SOP 通常包含大量文本内容。考虑到 LLM 在处理长输入时效果往往显著下降 [40]，直接在如此长的文本上进行推理对模型构成重大挑战。

**挑战三：推理效率低。** LLM 的推理效率低——典型的 LLM（如 ChatGPT）生成一次响应通常需要数秒。在实际部署场景中，这一延迟远非理想，尤其在故障事件中产生大量告警时尤为突出。

### 2.6 本文方案概述

针对上述挑战，本文提出 COLA，一种融合关联挖掘与 LLM 推理的新型混合在线告警聚合方法。COLA 由两个模块组成：关联挖掘模块和 LLM 推理模块。

为应对**挑战一（缺乏领域知识）**，我们提出利用上下文学习（In-Context Learning, ICL）向 LLM 提供领域特定知识。ICL 是一种提示词工程方法，通过在提示词中展示相关示例，使 LLM 能从示例中学习并执行任务，已被证实在多种软件工程任务中有效 [16, 41]。

为应对**挑战二（长文本理解）**，我们提出多轮提示机制，逐步对 SOP 中的长文本进行摘要，提取关键知识以支撑告警间关系的推理。

为应对**挑战三（推理效率低）**，我们在框架中引入了一个轻量级且高效的关联挖掘组件。该组件旨在过滤出具有高置信度的关联告警对，仅将不确定的告警对送入 LLM 推理模块进行深入分析。由此，LLM 仅需处理少量不确定的告警对，从而大幅提升了整体效率。

为评估所提方法，我们从 Cloud X 的生产环境中收集了三个数据集，包含 500,000 条告警和 3,000 份 SOP。大量实验表明，COLA 的 F1 分数达到 0.901 至 0.930，显著优于所有最先进的基线模型。此外，尽管引入了计算密集型的 LLM，COLA 仍展现出与现有方法相当的效率。

### 2.7 主要贡献

- 我们是首个提出在告警聚合中引入详细知识（即 SOP）的工作。通过利用这些广泛的知识，我们能够有效聚合语义差异大或出现频率低的告警（第 2 节）。
- 我们提出 COLA——一个包含关联挖掘组件和 LLM 推理组件的混合框架，专为在线告警聚合而设计。该混合方法使我们能够充分利用 LLM 的能力，同时保证在实际场景中高效处理大量告警（第 3 节）。
- 我们在工业界生产环境中收集的三个数据集上评估 COLA。评估结果表明，COLA 以 0.901 至 0.930 的 F1 分数超越最先进方法，同时保持可比的效率（第 4 节）。
- 我们已将 COLA 部署在 Cloud X 的生产环境并运行四个月，在本文中分享实践经验以惠及社区（第 5 节）。

## 三、Section 2: BACKGROUND AND MOTIVATION 翻译

### 3.1 告警与 SOP (2.1 Alerts and SOPs in Cloud X)

**告警 (Alert)**。当监控数据中检测到异常或由客户上报时，系统产生告警 [39]。告警用于通知值班工程师及时处理异常，并具有多种属性，包括告警 ID、标题、创建时间、到达时间、处置时间、所属服务、物理区域和负责工程师等。ID 对每条告警唯一，标题可视为告警的摘要。创建时间、到达时间、处置时间分别表示告警被触发、被值班工程师接收、被值班工程师处置的具体时刻。服务属性指示告警的所属服务，区域属性指示部署该服务的服务器所在的物理位置。

**SOP**。标准操作程序文档 [38] 是由工程师在 Cloud X 中维护的告警描述文档。这些文档以自然语言编写，辅以截图、表格和网站链接，为告警处理提供参考。SOP 包含 ID、标题、严重级别、告警的详细解释、对系统的影响、可能原因和推荐处置步骤。ID 是标识符，标题简要介绍异常情况。严重级别指示处理的优先级。解释部分详细阐述有关此告警的更多信息。影响部分展示该告警对所属服务和整个系统造成的后果。可能原因和推荐处置步骤由工程师从类似历史告警中总结得出。此外，云服务的领域知识也隐含地蕴含在工程师撰写的注释和解释中。然而，SOP 篇幅长且非结构化，因此自动提取关键信息并进一步分析其关联关系具有相当挑战。

### 3.2 动机示例 (2.2 Motivating Example)

我们以 Cloud X 中 2023 年 1 月发生的一次真实服务故障作为动机示例。该故障的根源是某存储服务器中固态硬盘 (SSD) 电压过低（告警 1），导致对象存储设备 (OSD) 进程中断。具体地，由于 SSD 供电不足，部分 OSD 的 I/O 操作未能完成或被阻塞（告警 2），这一异常被 OSD 进程监控器捕获（告警 3），进而导致一个 OSD 节点被标记为不健康（告警 4）。随后，存储集群的健康监控器也上报了这一异常（告警 5）。最终，在应用层，我们观测到 Redis 服务的成功连接数下降（告警 6）。

将这些海量告警进行关联对值班工程师构成重大挑战，因为这需要丰富的经验和系统理解，才能解读来自不同层级、不同服务的告警。在本例中，告警 1-6 覆盖了基础设施层的硬件、平台层的节点和进程，以及应用层。已有研究依赖统计相关性或语义相似度来进行告警聚合，但这些方法在此情况下表现不佳：对于表 1 中的告警，并非所有告警都有足够的历史出现量以支撑统计分析（如告警 5 和 6 较为罕见），同时告警 1-6 之间的文本语义相似度极低。

我们提出基于每条告警的知识（即 SOP）进行推理，以实现更有效的告警聚合。云系统中与每条告警关联的 SOP 文档详细而全面，篇幅约为 3-4 页 A4 纸。受 LLM 在各类挑战性任务上最新突破的启发 [17]，我们旨在利用 LLM 来理解这些 SOP，提取关键信息，并基于推理进行告警聚合。通过这种方式，我们可以基于广泛的知识链接告警，并从 LLM 获得可解释的结果。值得注意的是，这种方法能够利用分布在云系统不同服务和层级中的知识，执行更为有效的告警聚合。

### 3.3 挑战 (2.3 Challenges)

尽管 LLM 具备从自然语言中提取知识并进行推理的能力，在实际应用中仍存在三个挑战：

**挑战一：低效率。** 在实际工业场景中，LLM 生成响应非常耗时。在 Cloud X 中，LLM 总结一份 SOP 需要超过 10 秒，进行推理则需要更长的时间。当服务故障发生时，大量告警被触发，不可能将所有告警输入 LLM 进行及时处理。为克服低效率挑战，我们首先用关联挖掘模块评估相似度分数对告警进行过滤，丢弃具有正相似度分数的告警对，LLM 模块仅需处理少量不确定的告警。

**挑战二：长文本理解。** LLM 的上下文理解和总结能力随着文本长度的增加而下降。大多数 LLM 的输入长度有限。尽管部分 LLM 声称支持无限输入长度，但同时强调在输入数千字后效果会下降。在我们的设计中，LLM 的提示词包含两条告警及其 SOP，通常超出这些限制。为克服长文本理解挑战，受 Chain-of-Thought (COT) [37] 设计的启发，我们提出与 LLM 进行两轮交互：在第一轮请求中，要求 LLM 提取并总结 SOP 中的知识；在第二轮中，要求 LLM 基于总结后的 SOP 对告警进行分析和推理。

**挑战三：缺乏领域知识。** LLM 是一个通用语言模型，缺乏在线服务系统的领域知识。LLM 使用互联网上大量可访问的数据进行训练，然而在线服务系统运维的知识（尤其是内部术语和描述）大多未公开。因此，LLM 难以对告警的因果分析生成合理的答案。为克服领域知识缺乏的挑战，我们采用两种技术来利用从 SOP 中总结的知识：上下文学习（ICL）和监督微调（SFT）。

## 四、COLA Introduction 的行文结构分析

COLA 论文的 Introduction 遵循一套非常清晰的叙事逻辑链：

```
问题定义 (告警风暴 → 人工处理不可行)
    ↓
现有方案及其具体缺陷 (语义方法 + 统计方法, 各举2-3个具体系统 + 具体反例)
    ↓
本文思路 (引入外部知识 SOP + LLM 推理, 一句话讲清直觉)
    ↓
面临的挑战 (3个, 编号, 每点2-3句)
    ↓
本文方案 (对应3个挑战逐一给出解决思路)
    ↓
系统数据 (数据集规模、F1分数)
    ↓
四大贡献 (编号列表)
```

然后 Section 2 用**一个具体真实案例**（SSD电压低 → OSD中断 → 6条跨层告警）把上述所有抽象概念全部具象化。

### 核心特征

1. **第一句就进入问题**, 不写 "With the rapid development of..." 式铺垫
2. **每段都有具体数字或名称**: Amazon $100M/hr, Cloud X, 500,000 alerts, 3,000 SOPs, F1 0.901-0.930
3. **每指出一类方法的缺陷, 必给具体例子**: "server overload" vs "database slowdown" 语义不同但因果关联
4. **挑战与方案一一对应**: 三个挑战 → 三个解决思路, 结构严丝合缝
5. **真实故障案例驱动**: Section 2 完整演示一遍 "根因如何引发跨层告警", 让读者直观理解问题
6. **短句为主**: 英文原文每句平均 25-30 词, 一句一个意思
7. **不用 "不仅如此"、"更重要的是" 等递进连词**, 而是用 "However"、"Specifically"、"For example" 等精确连接
