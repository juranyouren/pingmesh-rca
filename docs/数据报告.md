# Pingmesh RCA 数据报告

> 面向方法改进的数据驱动分析。不是工作报告——是数据说了什么、我们试了什么、还有哪些方向没有被数据充分验证。

---

## 一、数据集概述

### 1.1 数据来源

所有数据来自华为云生产环境。Pingmesh 拨测系统在检测到端到端丢包后，触发一次故障诊断流程，系统自动拉取：

- **全链路拓扑**：此次故障涉及的所有网络设备的物理连接关系（Spine-Leaf CLOS 架构），每台设备记录 `role`（SPINE/LEAF/CORE）、`mgmt_ip`、`linked_from`（上游邻居 IP 列表）、`linked_to`（下游邻居 IP 列表）、`cross`（路径交汇次数）
- **告警列表**：每台设备在故障时间窗口内的告警记录（alarm_name、alarm_time、alarm_ip_ad）
- **日志列表**：设备系统日志（syslog），格式同告警
- **故障元信息**：`info.json`，包含 source_ip（丢包源）、sink_ip（丢包目的）、alarm_time（Pingmesh 触发时间）、scenario_code（POD NETWORK 等）
- **根因标注**：运维工程师人工标注的根因设备 IP（`label.json`，含 `abnormal_node` 列表）

### 1.2 可用数据集

| 数据集 | 路径 | 案例数 | 标注方式 | 告警完整度 | 状态 |
|--------|------|--------|---------|-----------|------|

| `nodes_max_labeled` | data/node/nodes_extend | **159** | 人工 | **完整**（告警恢复后） | **当前主力** |

```

│   │       └── pingmesh-{csn}-全链路.json  # 设备节点列表
```

### 1.4 如何使用

**切换数据集**：设置环境变量 `PINGMESH_DATA` 或修改 `scripts/common.sh`。

```bash
# 使用 159 例完整数据
export PINGMESH_DATA=/home/sbp/lixinyang/pingmesh/data/node/nodes_max_labeled

# 跑消融
./scripts/run_full_ablation.sh

# 跑 LLM 推理
./scripts/run_inference.sh my_test

# 评分
python Sys/Score/Score_N.py data/res/my_test/res.json
```

**预处理新原始数据**：

```bash
python Sys/Preprocess/Preprocessor.py \
  --raw data/raw/pingmesh_v1 \
  --out data/node/nodes_v1 \
  --write
```

## 二、已进行的尝试与结果

### 2.1 全景时间线

| 阶段 | 数据 | 核心发现 | Top-1 |
|------|------|---------|-------|
| 毕设 v1.0 | 104 例（非人工标注） | 无向 PageRank + LLM CoT 推理，存在多处方法论缺陷 | 60% |
| 泄漏发现 | 146 例 | temporal_score 从 label.json 读根因设备时间戳，76% 是泄漏 | 87%(虚假) → **57%** |
| 泄漏修复后 | 146 例（脱敏） | 告警稀疏卡住时序：80% case ≤5 台设备有告警 | **56.64%** |
| 当前 | **159 例**（完整告警） | 告警恢复后数据质量跳变，远高于脱敏数据 | **76.10%** |

### 2.2 消融实验（方法维度的分解）

**数据**：159 例人工标注 `nodes_max_labeled`  
**方法**：pure algorithm（不依赖 LLM），Topo + Temporal 等权融合

| 组合 | Top-1 | Top-3 | Top-5 | 含义 |
|------|-------|-------|-------|------|
| [1,2] topo+temporal (manual 权重) | **76.10%** | 85.53% | 91.19% | 当前天花板 |
| [2] temporal only | 62.89% | 88.05% | 94.34% | 时序是最强单信号 |
| [1] topo only | 50.31% | 74.21% | 84.28% | 拓扑信号独立有效 |

**已确认的结论**：

1. **时序 + 拓扑是真实协同，不是泄漏**。二者独立贡献分别为 63% 和 50%，融合 76%。+13pp over temporal——互补证据确凿。
2. **告警权重来源差异不大**。manual 权重（76.10%）vs LLM 学习权重（66.67%），差异 9pp 但 fused 后接近。告警权重对 PR 的影响在完整数据上被淹没了（告警多了之后 `len(alarms)*2.0` 数量分支主导 personalization）。
3. **Top-1 到 Top-5 提升了 15pp**（76%→91%）。剩余 9% 的 case 中根因完全不在 Top-5——这是真正的硬骨头。

### 2.3 LLM 角色的实验

| 实验 | 方法 | Top-1 | 结论 |
|------|------|-------|------|
| 后置重排（旧 prompt） | 让 LLM "审核并重排" | 87%→33% | LLM 缺乏约束 + 脱敏数据告警空 → 主动破坏正确排名 |
| 后置重排（新 prompt） | "默认信任算法" | 76.10%→75.47% | LLM 应在信号接近时裁决，其余默认可信算法 |
| 前置分类（global taxonomy） | 全量告警 causal/symptom/noise | temporal +1.9pp, fused -6.6pp | 分类预测不准时反噬 |
| 前置分类（per_case） | 逐 case 分类 | 待评估 | — |

**LLM 的核心价值在哪里**：数据表明，LLM 不应该被用作"从排序中再挑一遍"的重排器——当 Top-1 和 Top-2 的综合分差 >15 时，LLM 改错的概率高于改对的。LLM 的真正价值在于**前置信号增强**（告警分类）和**失败 case 的针对性分析**，而非全局后置重排。

### 2.4 失败案例分析

对 Top-1 失败 case 的 node 数据分析揭示了三类主导失败模式：

| 失败模式 | 占比估计 | 特征 | 可能的解法 |
|---------|---------|------|-----------|
| **告警量碾压** | ~40% | rank1 告警量是 gt 的 5-68 倍，temporal density 被刷量 | 密度改用去重种类数而非原始计数 |
| **综合分接近** | ~25% | gt 与 rank1 差异 <0.05，算法随机排序 | LLM 裁决（这些正是 LLM 该介入的 case） |
| **时序/拓扑误导** | ~35% | 单一信号过强把 gt 挤出 | 自适应融合权重（per case 调参） |

### 2.5 已弃用的方向

| 方向 | 原因 |
|------|------|
| co_occur Skill | 纯规则告警匹配在脱敏数据上单独 Top-1 ≈ 1-8%，等权融合拖后腿 |
| Modifier 拓扑剪枝 | 把全链路 300+ 节点压缩到 K=10，根因 IP 经常被剪出候选集（82/104） |
| 有向/无向 PageRank 对比消融 | dir 和 undir 差异不显著，Spine-Leaf 中方向标签不编码因果 |
| 告警风暴叙事 | 脱敏后无法验证，且新数据不再按告警量分桶 |

## 三、数据告诉我们的可能性

### 3.1 天花板估计

当前 76% Top-1 能在多大程度上改进？分解剩余 24% 的 38 个失败 case：

| 失败类型 | case 数 | 可救回吗？ |
|---------|--------|-----------|
| 根因完全不在 Top-5（miss 桶） | ~13 | 困难——需要从根本上改变算法排名，而非调参 |
| 根因在 Top-3（rank 2-3） | ~14 | 容易——后置 LLM 裁决或自适应融合权重 |
| 根因在 Top-5（rank 4-5） | ~11 | 中等——Top-5→Top-1 需要更强的区分信号 |

**可救回的上限**：~15-20 个 case（in_top3 + in_top5 桶），可推至 85-89%。miss 桶需要改数据或新方法（multi-modal、时序精细建模）。

### 3.2 数据中尚未被利用的信号

1. **告警描述文本**：JSON 中每条告警有 `alarm_description` 字段（如 "CID=0x807a040c-alarmID=0x08520003;The interface status changes. ifName=100GE1/1/25, AdminStatus=UP, OperStatus=DOWN, Reason=Interface physical link is down"）。这个文本包含了**具体的故障端口、原因、物理链路**——目前完全没有被用到。即使是不懂网络的 LLM 也能读懂 "OperStatus=DOWN" 和 "physical link is down" 的含义。**这是数据中最大的一块未开采金矿**。

2. **设备角色区分**：SPINE/CORE/LEAF 在 CLOS 架构中有不同的故障传播特性。一台 SPINE 故障会同时影响大量 LEAF。目前角色信息只在 evidence table 中展示给 LLM，算法层面只把 role 作为字符串，没有针对角色建模。

3. **告警级联的时间差**：同一条故障链路上，上游设备第一条告警和下游设备第一条告警之间有时间差。如果这个时间差存在稳定的模式，可以用于区分因/果。目前只用了 absolute time（earliest/first），没有用 relative time delta。

4. **source_ip/sink_ip 的拓扑意义**：Pingmesh 检测到的是端到端丢包，source_ip 和 sink_ip 是受影响的端点。从端点出发做双向 BFS，可以找到"距离加权"候选集——越靠近端点的设备越可疑。目前这个信号只作为 personalization 的弱加分项（+0.5），没有被充分利用。

### 3.3 未覆盖的实验空间

1. **超参扫描**：PageRank α（当前 0.85）、时序窗口（当前 5min）、融合权重（当前 0.5/0.5）、Top-K（当前 5）均未做敏感性分析。如果某些参数对结果高度敏感（例如 α 从 0.85→0.5 导致 topo 单独掉 10pp），说明当前取值是巧合最优，需要交叉验证。

2. **temporal 精细建模**：Burst/EarlyBird/Density 三特征的权重（0.40/0.35/0.25）是手动设定的。如果按 case 属性（告警量、拓扑规模）自适应调整，可能提升。

3. **自适应融合**：当前等权融合在所有 case 上用同一策略。失败 case 的分析显示，有些是 topo 正确 temporal 错误，有些反之。简单加权不能同时应对两类。动态调权（如按 case 中告警设备的比例决定 temporal 权重）可能同时解决两类失败。

4. **跨数据集验证**：NIKA（640 例 CLOS 拓扑）、或公司内部其他 data center 的数据集。目前为止所有消融都是 159 例单数据集——论文中缺少泛化实验。

## 四、数据质量注意事项

1. **label.json 中的 abnormal_node 需要在 preprocess 后手动筛选**：当前 rootcause_analysis 的全部条目都被写入了 label.json（含多个 ranking），尚未人工缩减到真正的根因。Score_N 的 `_get_groundtruth` 取 ranking 前三条的 IP，可能存在多条 gt 中含有非根因的情况。这对 Top-1 影响不大（只要有一个真根因在前三即可），但 Top-3/Top-5 的计算略微宽松。

2. **告警权重表的覆盖率**：`all_alarms.json` 中仅覆盖了人工总结的高危告警类型，大量实际出现的告警名称未命中权重表。这导致 PageRank personalization 在大部分设备上由 `len(alarms) * 2.0` 数量分支主导——即 PR 更接近"告警最多的设备"而非"告警最危险的设备"。**这是当前 PR 区分度低的结构性原因**。

3. **LLM 推理路径使用的评分函数和消融路径一致**：已通过 `compare_all.py` 验证全量 match。后续不再需要反复交叉验证。
