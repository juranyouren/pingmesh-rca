# 基于拓扑-时序证据融合与信任门控的 Pingmesh 根因定位方法

## 1. 论文定位

本文研究数据中心网络中由 Pingmesh 告警触发的根因设备定位问题。Pingmesh 能可靠发现端到端异常，但在 ECMP、高扇出 Clos 拓扑和告警级联场景下，无法直接指出物理根因设备。

论文主张不是让大模型直接完成网络 RCA，而是先用确定性算法生成可信 Top-K 候选，再用可解释信任门控决定是否需要 LLM 仲裁。

推荐题目：

> 基于拓扑-时序证据融合与信任门控的 Pingmesh 根因定位方法

## 2. 核心问题

给定一个 Pingmesh 故障 case：

- `info.json`：source、sink、故障触发时间和场景信息。
- 全链路拓扑：设备角色、管理 IP、上下游连接和路径交汇信息。
- 设备告警和日志：告警名称、时间、设备 IP 和描述文本。
- `label.json`：人工根因标注，仅用于离线评测。

目标是在不读取标签的推理路径中输出根因设备 Top-K IP 列表，并用 Top-1、Top-3、Top-5 命中率评估。

## 3. 方法总览

整体流程：

```text
Pingmesh case
  -> 数据预处理
  -> Topology Ranker
  -> Temporal Ranker
  -> Evidence Fusion
  -> Trust-Tree Gate
  -> 自动接受 / LLM 仲裁 / 人工复核
  -> Score_N Top-K Evaluation
```

### 3.1 拓扑排序器

拓扑排序器建模物理网络结构和告警分布，回答“从拓扑传播角度看，哪些设备更像根因”。

主要证据：

- 有向拓扑 PageRank。
- source/sink 邻近性。
- cross-path 交汇次数。
- 告警权重和告警数量。
- 设备角色，如 SPINE、LEAF、CORE。

该模块提供结构性证据，但在 Clos 高扇出拓扑中容易受到中心节点和拓扑稀释影响，因此不能单独作为最终可信判断。

### 3.2 时序排序器

时序排序器建模告警与故障触发时间之间的关系，回答“从异常发生时间看，哪些设备更像根因”。

主要证据：

- burst：故障窗口内告警爆发程度。
- early-bird：是否更早出现异常。
- temporal density：故障时间附近的告警密度。

当前实验显示，时序是最强单信号，但它会被告警量大的症状设备误导。

### 3.3 多证据融合

拓扑和时序是互补证据。当前 159 例数据上的结果表明：

| 方法 | Top-1 | Top-3 | Top-5 |
| --- | ---: | ---: | ---: |
| topo only | 50.31% | 74.21% | 84.28% |
| temporal only | 62.89% | 88.05% | 94.34% |
| topo + temporal | 76.10% | 85.53% | 91.19% |

融合后 Top-1 相比 temporal only 提升约 13 个百分点，说明拓扑和时序不是重复信号。

### 3.4 Trust-Tree Gate

信任门控是本文的关键机制。它不使用连续置信分数，而是使用可解释规则树分别判断拓扑证据和时序证据：

- `strong`
- `uncertain`
- `weak`

路由逻辑：

```text
if topo == weak and temporal == weak:
    operator_review
elif topo 和 temporal 排名接近:
    accept_combined
elif temporal == strong and topo != strong:
    accept_temporal
else:
    invoke_llm
```

该机制的论文价值：

1. 避免 LLM 破坏高置信确定性排序。
2. 降低不必要的 LLM 调用成本。
3. 对弱信号 case 显式进入人工复核。
4. 每个自动决策路径都能解释。

### 3.5 受控 LLM 仲裁

LLM 不作为主排序器，而作为受门控约束的语义仲裁器。它只处理：

- 拓扑和时序冲突的 case。
- Top-1/Top-2 分数接近的 case。
- 告警描述中存在明确语义证据的 case。
- 确定性算法无法高置信接受的 case。

提示词应要求 LLM 默认信任算法排序，只有在候选证据提供明确反证时才调整排名。

## 4. 预期创新点

### 创新点 1：面向 Pingmesh 的拓扑-时序多证据根因排序框架

将 Pingmesh 端到端异常、Clos 拓扑、设备告警时序和告警权重统一为设备级 Top-K 根因定位问题。

### 创新点 2：可解释 Trust-Tree 信任门控

提出拓扑信任树和时序信任树，按证据强弱将 case 路由到自动接受、LLM 仲裁或人工复核路径。

### 创新点 3：受控 LLM 仲裁机制

证明全量 LLM 重排不是稳定收益来源，并将 LLM 限制在冲突与低置信 case 中，提升工程可控性。

### 后续增强点：告警描述语义利用

`alarm_description` 中包含端口状态、物理链路 down、接口原因等语义信息。后续可将其作为 LLM 仲裁和算法加权的新证据。

## 4.5 文献支撑补充

Trust-Tree Gate 可放入 selective prediction / abstention / confidence estimation 的研究脉络中理解：高置信 case 自动输出，低置信或冲突 case 交给更强但更昂贵的 LLM 或人工复核。具体文献整理见 `docs/文献调研补充_信任门与LLM角色.md`。

LLM 在本文中的定位也应与现有 LLM-RCA 工作区分开：RCACopilot、RCAgent、TAMO、BiAn 等证明 LLM 可用于运维文本理解、工具调用和辅助定位；OpenRCA 以及 LLM reasoning failure 研究则说明 LLM 在复杂 RCA 中存在证据选择、长上下文和多跳推理不稳定。本文的贡献是将 LLM 限制为受门控约束的仲裁器，而不是无约束主排序器。

## 5. 实验矩阵

### 实验 1：确定性排序消融

脚本：`scripts/run_paper_01_skill_ablation.sh`

比较：

- topo only
- temporal only
- topo + temporal
- manual 权重 vs LLM-learned 权重

目标：证明拓扑和时序的互补性。

### 实验 2：Trust-Tree 路由效果

脚本：`scripts/run_paper_02_gate_routing.sh`

比较：

- deterministic pipe
- gate_eval
- gate_pipe

目标：统计自动接受、LLM 路由和人工复核比例，以及不同路由下的 Top-K 表现。

### 实验 3：LLM 仲裁有效性

脚本：`scripts/run_paper_03_llm_arbitration.sh`

比较：

- pure skill
- full LLM reranking
- gated LLM arbitration

目标：验证门控式 LLM 是否比全量 LLM 更稳定，并统计 LLM 调用节省比例。

### 实验 4：门控策略与选择分析

脚本：`scripts/run_paper_04_gate_policy_analysis.sh`

比较：

- baseline
- strict_combined
- conservative

目标：分析不同门控策略对准确率、调用成本和人工复核比例的影响。

### 实验 5：候选摘要缓存实验

脚本：

- `scripts/run_paper_05_precompute_summary_cache.sh`
- `scripts/run_paper_06_cached_summary_llm.sh`

目标：验证小模型候选摘要能否减少主 LLM prompt 成本，同时保持 RCA 效果。

## 6. Baseline

当前保留以下 baseline：

- TraceRCA：基于 trace/路径相似性的传统 RCA。
- NetEventCause：无拓扑事件因果 baseline。
- BiAn：LLM failure localization baseline。

脚本：`scripts/run_baselines.sh`

baseline 用于说明本文方法在数据中心 Pingmesh 场景下的定位能力和工程效率。

## 7. 论文结构建议

1. 绪论：Pingmesh 异常检测与根因定位缺口。
2. 相关工作：Pingmesh、DCN RCA、拓扑诊断、告警时序分析、LLM for AIOps。
3. 问题定义：case 输入、Top-K 输出、评价指标和数据泄漏约束。
4. 方法设计：拓扑排序、时序排序、融合、Trust-Tree Gate、LLM 仲裁。
5. 实验评估：主结果、消融、门控、LLM 成本、baseline。
6. 失败案例分析：告警量碾压、排名接近、拓扑误导、根因不在 Top-K。
7. 总结与展望：告警描述语义增强、跨数据集验证、自适应融合。

## 8. 当前边界

- 内部生产数据不可公开。
- 外部 LLM API 不进入实验路径。
- 推理路径不能读取 `label.json`。
- LLM 只作为受控仲裁器，不作为无约束主排序器。
